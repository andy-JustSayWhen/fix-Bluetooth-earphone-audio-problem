## 对蓝牙耳机音质异常排查的意义

Bose QC Headphones 当前承担音频输入，而且输入采样率是 16 kHz；它同时承担系统提示音输出，QQ音乐也实际从 Bose 播放。目标蓝牙耳机同时承担播放和麦克风输入时，应用可能触发蓝牙通话模式，从而出现空洞、失真、低码率、传输不稳定或没有声音。你已经确认此时存在音质异常，因此后续排查重点应放在 QQ音乐的应用级输出路由、Bose 麦克风是否被占用，以及蓝牙设备是否切换到低质量通话链路。

后续遇到“声音空洞、失真、码率很低、传输不稳定或没有声音”时，应同时记录：

1. 系统默认普通播放输出是什么设备。
2. SoundSource 或其他音频路由工具是否为出问题的应用设置了专属输出。
3. 出问题的应用实际从哪个设备播放，不能只看系统设置里的默认输出。
4. 系统输入是否启用了目标蓝牙耳机麦克风。
5. 出问题瞬间目标蓝牙设备的采样率和连接状态。

## 语音输入后复现：真正触发变量是 Bose 麦克风被选为输入

### 对照过程

用户在当前设置下完成一次语音输入，QQ音乐从暂停中自动恢复后，输出仍然是 Bose，但音频异常重新出现。随后只通过顶部菜单栏切换输入设备：

| 状态 | 输出 | 输入 | QQ音乐听感 |
| --- | --- | --- | --- |
| 语音输入后 | Bose QC Headphones | Bose QC Headphones | 空洞、失真、低码率或传输异常 |
| 切换输入后 | Bose QC Headphones | Redmi 电脑音箱 | 恢复正常 |

关键点是：两张截图中的输出设备始终都是 Bose，只有输入设备发生变化；切换输入后异常立即消失。因此“输出设备选错”不能解释现象，“Bose 作为输入时触发了另一种蓝牙音频模式”可以解释整个对照结果。

证据图片：

- [Bose 同时作为输出和输入时异常](evidence/voice-input-bose-input-abnormal.png)
- [输出仍为 Bose、输入切到 Redmi 后恢复](evidence/voice-input-redmi-input-normal.png)

### 根因判断

Bose QC Headphones 的蓝牙服务同时包含 HFP（蓝牙通话模式）和 A2DP（蓝牙立体声播放模式）。当 Bose 麦克风被选为系统输入时，macOS 需要建立耳机的双向音频链路，很可能把播放链路切入 HFP 兼容路径；这条路径的带宽和声道能力明显低于普通立体声播放，所以听起来会空洞、失真、像低码率或不稳定。

当输入切换到 Redmi 电脑音箱后，Bose 麦克风不再是默认输入，Bose 只承担输出，音频链路可以回到 A2DP 立体声播放，异常随即消失。

因此当前高置信结论是：

> 根因不是 QQ音乐把输出切到了错误设备，也不是单独的 44.1 kHz / 16 kHz 数值不匹配；更上游的触发事件是某个应用调用了蓝牙耳机麦克风，使系统从 A2DP 立体声播放模式切到 HFP/HSP 双向通话模式。麦克风调用结束后，系统或应用没有正确恢复 A2DP，后续继续播放才出现异常。Bose 仍被选作输入是当前复现条件，但不是完整根因。

### 证据边界

- 行为对照已经直接证明“输入为 Bose”是异常的触发条件。
- 系统蓝牙查询确认 Bose 同时支持 HFP 和 A2DP，支持该机制解释。
- `system_profiler`（macOS自带的系统信息查询工具）显示的是设备能力和当前默认路由，不能单独证明某一瞬间已经使用哪一个蓝牙协议；若要把 HFP/A2DP 切换做成直接证据，需要在异常出现的瞬间采集音频引擎的实际声道数和活动采样率。

### 排查结论

目前无需继续追查 QQ音乐输出设备：QQ音乐一直在 Bose 上播放。后续排查应围绕“应用调用蓝牙麦克风后，蓝牙播放模式是否成功恢复”展开，而不是只检查语音输入结束后 Bose 是否仍是默认输入。

## 跨平台复现规律

用户补充了 Windows 和 macOS 上都出现过的相同规律：

1. 蓝牙耳机只作为输出设备使用时，音乐和视频播放正常。
2. 某个应用调用这副耳机的麦克风。
3. 麦克风调用结束后，再次通过同一副耳机播放，声音变得空洞、失真、低码率或不稳定。
4. 将输入切换到其他设备，或让系统重新建立音频路由后，听感恢复。

这说明项目要排查的通用问题应命名为“蓝牙耳机麦克风调用后的播放模式未恢复”，而不是“蓝牙耳机被选作输入导致音质异常”。后者只是当前机器上最容易触发和解除问题的操作。

## macOS 解决方案边界

当前 Bose QC Headphones 的蓝牙设备信息为：

```text
device_services: HFP AVRCP A2DP Braille ACL
device_firmwareVersion: 1.0.6
```

这说明当前耳机公开的是经典蓝牙的 HFP（蓝牙双向通话模式）和 A2DP（蓝牙立体声播放模式），没有公开 LE Audio（新一代低功耗蓝牙音频）或 LC3（新一代蓝牙音频编码）能力。

因此，在“同一副 Bose 耳机、同时使用 Bose 麦克风、同时保持正常立体声音乐”这三个条件都不改变的情况下，macOS 上没有可靠的软件修复。SoundSource、QQ音乐采样率选项或重新切换输入，只能触发链路重建，不能改变经典蓝牙的带宽和模式限制。

### 可行路线

1. **保持当前耳机并保证音乐正常**：Bose 只作为输出，语音输入使用 MacBook 麦克风或 USB 麦克风。代价是不用 Bose 自带麦克风。
2. **保持当前耳机并使用 Bose 麦克风**：进入 HFP 通话模式，接受音乐变成低质量或某些应用暂时无声；语音输入结束后把输入切回其他设备，让 Bose 恢复 A2DP。可以把这一步做成一键切换，但不能保证通话模式下仍是正常音乐音质。
3. **同时保留 Bose 麦克风和正常音乐**：更换支持 LE Audio 的耳机，或使用支持完整双向音频的 USB / 2.4 GHz 接收器方案。这个方案改变硬件链路，才能绕开当前 Bose 的经典蓝牙模式限制。

有线耳机不会出现同样问题，是因为它不需要在 A2DP 与 HFP 之间切换无线链路；播放和麦克风可以在同一条有线音频连接上同时工作。

## 实测监视：Bose 同时作为输入和输出

### 监视方式

### 这次实测能证明什么

这次证据把“采样率/声道变化”和“语音输入事件”区分开了：

1. **真正可直接观测到的模式变化发生在把 Bose 选为输入时。** 输出从 44.1 kHz、双声道变成 16 kHz、单声道；这符合经典蓝牙 HFP（蓝牙双向通话模式）的低带宽特征。
2. **语音输入结束后，系统没有把 Bose 输出恢复为 44.1 kHz、双声道。** 因此异常播放时，Bose 仍停留在低质量双向链路，而不是已经恢复普通立体声播放后又单纯发生了采样率不匹配。
3. **语音输入期间的“卡顿—静音—恢复”没有伴随新的设备级采样率跳变。** 这说明应用音频流的暂停、重连或恢复过程可能发生在音频设备之上，单靠设备列表和标称采样率监视不到应用内部的流状态。

当前最稳妥的因果描述应修正为：把 Bose 麦克风选为输入会让 macOS 将 Bose 输出置于 16 kHz、单声道的蓝牙双向模式；QQ音乐随后在这条模式下经历一次音频中断与恢复，恢复后继续以异常听感播放。不能再把“麦克风结束后才发生一次可见的设备模式切换”作为已证实事实。

### 证据边界与下一步

本轮监视确认了实际设备层的采样率和声道变化，但没有直接读取蓝牙协议名称、编码器码率或 QQ音乐内部音频流格式。若继续深挖，需要在同一实验中同时记录 QQ音乐进程的音频流格式、音频中断通知和 Bose 输入切换回其他设备时的恢复时间；当前设备层证据已经足以确认：Bose 麦克风和正常立体声播放在这条经典蓝牙链路上不能同时保持。

| --- | --- | --- | --- |
| 语音结束后，输入切到 Redmi 电脑音箱 | Bose | 可以正常点击播放 | 音频无异常 |
| 随后输入切到 DJI | Bose | 无法点击播放 | 播放启动失败 |
| 先切到 Redmi，再切到 DJI | Bose | 可以点击播放 | 能播放，但音频异常 |

这组结果说明，“输入切到其他设备”并不是单一的修复动作，而是会触发不同程度的音频会话重建：

1. 切到 Redmi 能让 Bose 回到正常播放状态，说明这一步确实能解除当前 Bose 双向蓝牙链路的异常。
2. 直接切到 DJI 后，QQ音乐连播放控件都无法启动，说明应用音频会话可能处于未完成释放或重建失败状态，问题已经超出单纯的采样率显示范围。
3. 先切 Redmi 再切 DJI，QQ音乐可以恢复播放但仍异常，说明“应用能否播放”和“播放链路是否恢复为正常立体声”是两个独立结果。

因此，当前需要记录的变量不应只有最终输入设备，还要记录完整切换路径：

```text
语音结束
  ├─ 输入 -> Redmi：可播放，Bose 听感正常
  ├─ 输入 -> DJI：不可点击播放
  └─ 输入 -> Redmi -> DJI：可播放，但 Bose 听感异常
```

这进一步支持一个更精确的判断：语音结束后，macOS、QQ音乐和蓝牙音频设备之间可能存在音频会话释放顺序或重新协商状态问题。Redmi 的中间切换能够重建一部分状态，但切换到另一副蓝牙麦克风后，系统仍可能让 Bose 保持低质量双向输出，或让 QQ音乐保留不匹配的音频流格式。后续实测必须把“输入切换路径”和“QQ音乐播放控件是否可用”一起记录，不能只记录最终选中的输入设备。

## 多次监视后的新结论：选择 Bose 输出会联动选择 Bose 输入

用户进一步确认：在系统界面把输出切换为 Bose 时，默认输入也会自动变成 Bose；只要发生这次联动，音频就必然异常。运行记录 `run-20260716-015223` 和 `run-20260716-014704` 对此提供了直接的设备层证据。

### 设备监视记录

在 `run-20260716-015223` 中，约 182.27 秒处，监视器同时记录：

```text
默认输入：Redmi 电脑音箱 -> Bose QC Headphones
默认输出：Redmi 电脑音箱 -> Bose QC Headphones
Bose 输出：44100 Hz / 2 声道 -> 16000 Hz / 1 声道
```

在约 186.53 秒处，输出从 DJI 切回 Bose 时，Bose 输出再次变为 16 kHz、单声道；约 0.26 秒后短暂恢复到 44.1 kHz、双声道，但默认输入仍然是 Bose。这解释了为什么某些切换顺序下可以点击播放，却仍然听感异常：设备状态可能短暂恢复，应用音频会话却没有稳定恢复。

### 系统音频日志记录

同一运行记录的 `unified-audio-log.txt`（系统音频日志）在输出切回 Bose 的瞬间记录了：

- QQ音乐因重新配置等待而放弃当前音频处理周期，并停止当前输出。
- QQ音乐的默认输出设备发生变化，系统主动插入约 1 秒静音。
- 蓝牙音频服务请求配置变化，并把当前输出切换到 `tsco`（蓝牙通话传输链路）。
- 最佳采样率被确定为 16 kHz，立体声标记为关闭。
- 可用通话编码器记录为 `CVSD`（传统窄带通话编码）和 `mSBC`（宽带通话编码）；立体声能力为关闭。
- Bose 的音频传输开始于 `tsco`，QQ音乐随后记录“输出流格式已改变”，并把默认输出采样率适配为 16 kHz。

这已经不是单纯根据“16 kHz、单声道”做的模式推断，而是系统内部日志直接记录了 HFP（蓝牙双向通话模式）传输建立、16 kHz 适配和立体声关闭。

### 结论修正

当前最准确的触发链路是：

```text
用户选择 Bose 输出
  -> macOS 自动把默认输入联动为 Bose
  -> 蓝牙音频服务建立 HFP / tsco 通话传输
  -> 输出变为 16 kHz、单声道，立体声关闭
  -> QQ音乐暂停、插入静音并重建输出流
  -> 恢复后可能正常、异常，或播放控件暂时失效
```

因此，之前“只要 Bose 作为输出、输入不是 Bose 就一定正常”的描述需要加上一个前提：必须确认切换输出时没有被系统自动把输入重新选回 Bose。真正稳定的正常基线是：Bose 输出保持 44.1 kHz、双声道，且默认输入不能是 Bose；一旦输出切换动作联动了 Bose 输入，就会触发异常链路。

# macOS 音频监视器

本工具用于复现“蓝牙耳机同时作为输入和输出，完成一次麦克风使用后，播放音频出现卡顿、静音或音质异常”的过程，并把证据留在本地。

## 启动

在项目根目录运行：

```sh
./tools/run-macos-audio-monitor.sh
```

默认监视 10 分钟。需要缩短或延长时，可以在启动前设置秒数：

```sh
MONITOR_DURATION=180 ./tools/run-macos-audio-monitor.sh
```

启动后，前台会持续显示：

- 默认输入、默认输出和系统提示音输出；
- 设备的当前采样率和声道数；
- 设备运行状态；
- 根据设备指标推断的“疑似蓝牙通话模式”或“疑似蓝牙立体声播放模式”；
- 设备路由或模式变化的关键节点；
- 每 5 秒一次的当前状态心跳。

## 建议操作顺序

1. 启动监视器，确认前台显示已出现初始状态。
2. 保持 Bose 为输出，先记录一次正常播放时的基线。
3. 把 Bose 切为输入。
4. 完成一次语音输入。
5. 等 QQ音乐自动恢复播放，记录“卡顿—静音—恢复”和听感变化。
6. 如果需要，切换输入到其他设备，观察前台是否出现采样率和声道恢复。
7. 按 `Ctrl-C` 结束，或等待默认的 10 分钟结束。

如果要排查“切换到不同麦克风后结果不一样”，建议每次都从同一个起点开始，并把完整路径记下来，例如：

```text
语音结束 -> Redmi
语音结束 -> DJI
语音结束 -> Redmi -> DJI
```

同时记录 QQ音乐是“可以点击播放”“无法点击播放”，还是“可以播放但音频异常”。最终输入设备相同，不代表音频会话经历了相同的重建过程。

## 本地输出

每次运行都会创建一个独立目录：

```text
artifacts/audio-monitor/run-YYYYMMDD-HHMMSS/
```

文件用途如下：

| 文件 | 内容 |
| --- | --- |
| `samples.jsonl` | 每 250 毫秒一条完整设备快照，包含设备标识、名称、厂商、连接类型、全局/输入/输出采样率、输入/输出声道、可用采样率、运行状态和默认路由标记 |
| `events.jsonl` | 初始状态、默认路由变化、采样率变化、声道变化、运行状态变化和结束状态；每条事件同时保留变化前后的完整快照 |
| `system-profiler-start.json` | 启动前 macOS 系统音频设备信息 |
| `system-profiler-end.json` | 结束时 macOS 系统音频设备信息 |
| `unified-audio-log.txt` | 运行期间 `coreaudiod`（macOS 音频核心服务）和 QQ音乐相关的系统日志，能否产生内容取决于系统日志权限和当时日志级别 |
| `processes-start.txt` / `processes-end.txt` | 运行前后进程列表，用于确认 QQ音乐、音频路由工具和音频服务是否存在 |

前台只显示关键节点，详细快照不会全部刷屏，而是持续写入 `samples.jsonl`。这个文件是逐行 JSON（每行一条独立记录），即使监视器中途被终止，已经写入的部分仍然可以读取。

## 如何解释模式

macOS 公开的设备属性通常不会直接告诉我们当前蓝牙协议名称，所以工具不会把推断写成绝对事实：

- 蓝牙输出约 44.1/48 kHz 且为双声道，同时没有蓝牙输入声道：标记为“疑似蓝牙立体声播放模式”；
- 蓝牙输出约 16 kHz 且为单声道，同时存在输入声道：标记为“疑似蓝牙通话模式”；
- 其他组合：标记为“蓝牙模式未能仅凭设备指标判断”。

这样既能保留采样率、声道这些真实数据，也不会把设备指标推断冒充为直接读取到的蓝牙协议。

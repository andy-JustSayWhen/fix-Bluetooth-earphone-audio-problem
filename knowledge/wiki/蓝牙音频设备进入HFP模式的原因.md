## C1：Codex 语音输入后进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；macOS 15.7.7 / 24G720；环境记录号 `MACMINI9,1-175AC621`

输入设备：
`Redmi电脑音箱-3899`

输出设备：
`Redmi电脑音箱-3899`

问题现象：Codex 语音输入开始后，设备输出从高质量播放链路切入 16 kHz 单声道的 HFP 通话链路。

原因：Codex 实际读取目标蓝牙麦克风，系统为双向通话建立 HFP。

关键日志原文：

```text
22:18:48.718：Codex (Service) 向蓝牙声音插件注册。
22:18:48.947：Codex 开始读取 tsco 输入端点。
22:18:48.968：当前输出配置由 tacl 切换为 tsco。
22:18:56.643：语音结束后，系统开始断开该设备的 HFP 声音连接。
22:18:56.783：系统明确记录正在断开 SCO。
22:18:56.800：tsco 传输断开完成。
22:18:56.806：当前输出配置重新切换为 tacl。
22:18:56.909：A2DP 声音流重新开始。
```

退出 HFP 的方法：结束 Codex 麦克风读取，等待系统自行释放 SCO 通话链路。
本次恢复结果：`44.1 kHz / 双声道`、`tacl`，用户确认听感正常。该次属于正常释放，不是工具执行的强制恢复。

## C2：系统设置输入电平监测后进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；macOS 15.7.7 / 24G720；环境记录号 `MACMINI9,1-175AC621`

输入设备：
`Redmi电脑音箱-3899`

输出设备：
`Redmi电脑音箱-3899`

问题现象：打开系统设置“声音 -> 输入”并持续监测输入电平后，目标设备进入 HFP；切走输入后退出。

原因：系统设置为了显示输入电平，实际启动了目标蓝牙麦克风输入。

关键日志原文：

```text
HFPInputShim StartIO
```

退出 HFP 的方法：停止系统设置的输入电平监测，或切换到其他输入设备，等待系统释放 HFP。实测 2026-07-18 07:34:12 开始，07:34:17 切走输入后停止，07:34:19 输出恢复 `44.1 kHz / 双声道`。关键条件是页面确实启动了输入电平采集。

## C3：微信输入法提交格式请求后进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；macOS 15.7.7 / 24G720；环境记录号 `MACMINI9,1-175AC621`

输入设备：
`Redmi电脑音箱-3899`

输出设备：
`Redmi电脑音箱-3899`

问题现象：微信输入法没有真正开始录音，但设备从 A2DP 切入 HFP，输出变为 16 kHz 单声道。

关键日志原文：

```text
09:14:36.254：kBluetoothAudioDevicePropertyFormat request 0 -> 1
09:14:36.255：声音核心执行配置切换，选择 16 kHz、单声道通话端点
09:14:36.287：Current profile tsco
```

原因：微信输入法提交蓝牙输入格式请求，触发系统切换通话格式；该窗口没有同进程 `StartIO`。09:14:36.196 至 09:14:36.200 期间，`audioaccessoryd` 将 Redmi 设为临时默认输入；格式请求进程可反查为微信输入法 `com.tencent.inputmethod.wetype`。微信输入法为什么产生这条请求仍待确证。

退出 HFP 的方法：退出微信输入法，确认它没有自动重启，然后断开并重新连接设备。2026-07-18 09:22:32 起连续六次读取为输入 `16 kHz / 单声道`、输出 `44.1 kHz / 双声道`，日志持续显示 `Current profile tacl`。外置键盘唤醒、重复按键和快捷键转换只是尚未确证的上游线索。

## C4：蓝牙输入 A 导致蓝牙输出 B 进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；机型标识 `Macmini9,1`；硬件唯一标识 `175AC621-14A2-5E02-AEFD-F7D360137359`；macOS 15.7.7 / 24G720；蓝牙控制器 `BCM_4378`

输入设备：
`Redmi电脑音箱-3899`

输出设备：
`REDMI Buds 6 Pro 电竞版`

问题现象：选择蓝牙输入 A 后，另一台蓝牙输出 B 从 `tacl`、约 44.1 kHz、双声道切换为 `tsco`、16 kHz、单声道。

关键日志原文：

```text
08:36:04.139：耳机输出仍为 Current profile tacl
08:36:04.576：macOS 原生蓝牙声音服务发送设备配置变化
08:36:04.661：系统为耳机输出选择 16 kHz，立体声关闭
08:36:04.728：耳机输出变为 Current profile tsco
08:36:05.805：系统记录 1 个 SCO 设备、0 个 A2DP 设备
```

原因：在该主机、系统版本和设备组合中，macOS 原生蓝牙声音路由因“蓝牙输入 A + 蓝牙输出 B”组合而联动降级输出 B。关闭控制中心和声音设置页面后，HFP 仍持续到至少 08:39:42；没有观察到 A 的输入端开始采集，蓝牙链路日志持续显示 `AudioInput: 0`，SoundSource 已退出并停止应用级声音路由。

退出 HFP 的方法：把默认输入切回 USB、内置麦克风或其他非蓝牙输入，等待 B 恢复；必要时重新选择输出设备，让应用重建声音队列。切回 `MOONDROP Rays` 后，B 恢复 `tacl`、约 44.1 kHz、双声道，并重新出现 1 个 A2DP、0 个 SCO。重新提交相同输出路由、完整断开重连仍失败；其他方案尚未完成验收。

## C7：微信输入法把两个不同蓝牙端点放进同一声音会话后进入 HFP

宿主机：`andymacbook-air`；MacBookAir10,1；系统版本未在本实例摘要中固定

输入设备：
`DJI Mic Mini-9DC1E8`

输出设备：
`XIBERIA K03S`

问题现象：微信输入法语音输入短暂出现波形，随后耳机反复提示“已断开、已连接”，输入没有声音，XIBERIA 输出进入 HFP/HSP。

关键日志原文：

```text
15:28:17.939653  session: WeType(61338)
details:
  deviceUIDs:
    - 50-C0-F0-F3-6A-66:output
    - 58-B8-58-9D-C1-E8:input
  implicit_category: Record
  input_running: true
  output_running: false

There was an error setting the deviceUUIDs as there are more than one BT device connected = (
    "50:C0:F0:F3:6A:66",
    "58:B8:58:9D:C1:E8"
)
```

原因：微信输入法会话把一个蓝牙输出端点和另一台蓝牙设备的输入端点同时放入同一声音会话，macOS 拒绝这组双蓝牙端点配置，随后格式请求反复切换。失败窗口约为 2026-07-18 15:28:17 至 15:28:22；XIBERIA 输出在 16 kHz 和 44.1 kHz 之间反复切换。微信应用内置语音转文字使用 DJI 输入、BOSE 输出时成功，微信输入法输入输出都使用 XIBERIA 时也成功。

退出 HFP 的方法：本实例没有完成符合统一标准的自动恢复验证。已验证的绕过方式是把输出切到 MacBook 内置扬声器，或让输入输出使用同一蓝牙设备；不能把结束微信输入法写成已验证恢复方法。

## C8：Bose 输出被系统联动为输入后进入 HFP

宿主机：`MAC-home01-MacBookAir`；`andymacbook-air.local`；macOS 14.6.1

输入设备：
`Bose QC Headphones`

输出设备：
`Bose QC Headphones`

问题现象：Bose 从 `44.1 kHz / 双声道` 切为 `16 kHz / 单声道`，传统蓝牙通话链路建立；语音结束后 QQMusic 出现卡顿、静音和恢复异常。

原因：系统把 Bose 输出和输入联动到同一双向通话链路。

关键日志原文：

```text
Starting IO on profile tsco
最佳采样率 16000
stereo 0
可用 CVSD 和 mSBC
```

退出 HFP 的方法：将输入切换到 Redmi、Mac 内置麦克风或 USB 麦克风，同时保留 Bose 为输出。输入切到 Redmi 后，Bose 仍作为输出，QQMusic 可以继续播放，用户确认听感正常。

## C9：K03S 蓝牙输入输出切换后进入 HFP

宿主机：`MAC-home01-MacBookAir`；`andymacbook-air.local`；macOS 14.6.1

输入设备：
`XIBERIA K03S`

输出设备：
`XIBERIA K03S`

问题现象：K03S 从 `44.1 kHz / 双声道` 降为 `16 kHz / 单声道`；输入切走后可能短暂恢复又再次降级，QQMusic 出现输出队列停止、启动和控制延迟。

原因：蓝牙输入输出切换导致设备格式重新协商，应用输出队列重建与设备恢复不同步。

关键日志原文：

```text
tacl
tsco
QQMusic 输出队列停止、启动、再次停止、启动
```

退出 HFP 的方法：将 K03S 改用 2.4G 接收器，使系统把它识别为 USB 声音设备；或将输入切换到 Mac 内置、USB 或其他非目标蓝牙设备。2.4G 接收器模式下，监视器记录约 `48 kHz` 输出双声道、`48 kHz` 输入单声道，用户确认输入输出切换不再出现本实例异常。同一副 K03S 使用 2.4G 接收器时，多次输入输出切换均无异常；切回传统蓝牙后，输入输出都为 K03S 时再次出现 16 kHz 单声道和恢复异常。

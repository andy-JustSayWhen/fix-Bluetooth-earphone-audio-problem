## C1：Codex 语音输入后进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；macOS 15.7.7 / 24G720；环境记录号 `MACMINI9,1-175AC621`

输入设备：

- 名称：`Redmi电脑音箱-3899`
- 唯一标识：本条实测记录未留存蓝牙地址
- 状态：默认输入，并被 Codex 实际读取

输出设备：

- 名称：`Redmi电脑音箱-3899`
- 唯一标识：本条实测记录未留存蓝牙地址
- 状态：默认输出

问题现象：Codex 语音输入开始后，设备输出从高质量播放链路切入 16 kHz 单声道的 HFP 通话链路。

原因：Codex 实际读取目标蓝牙麦克风，系统为双向通话建立 HFP。

关键日志原文：

```text
22:18:48.718：Codex (Service) 向蓝牙声音插件注册。
22:18:48.947：Codex 开始读取 tsco 输入端点。
22:18:48.968：当前输出配置由 tacl 切换为 tsco。
22:18:56.643：语音结束后，系统开始断开该设备的 HFP 声音连接。
22:18:56.783：系统明确记录正在断开 SCO。
22:18:56.800：tsco 传输断开完成。
22:18:56.806：当前输出配置重新切换为 tacl。
22:18:56.909：A2DP 声音流重新开始。
```

退出 HFP 的方法：结束 Codex 麦克风读取，等待系统自行释放 SCO 通话链路。

恢复结果：本次实测恢复为 `44.1 kHz / 双声道`、`tacl`，用户确认听感正常。该次属于正常释放，不是工具执行的强制恢复。

结论边界：只能确认本机、本设备、本次语音输入成立；不能扩大为所有语音输入结束后都必然自动恢复。

## C3：微信输入法提交格式请求后进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；macOS 15.7.7 / 24G720；环境记录号 `MACMINI9,1-175AC621`

输入设备：

- 名称：`Redmi电脑音箱-3899`
- 唯一标识：蓝牙地址 `50:88:11:07:63:DA`
- 状态：被系统临时设为默认输入，但没有真正开始录音

输出设备：

- 名称：`Redmi电脑音箱-3899`
- 唯一标识：蓝牙地址 `50:88:11:07:63:DA`
- 状态：默认输出

问题现象：微信输入法没有真正开始录音，但设备从 A2DP 切入 HFP，输出变为 16 kHz 单声道。

原因：微信输入法提交蓝牙输入格式请求，触发系统切换通话格式；该窗口没有同进程 `StartIO`。

关键日志原文：

```text
09:14:36.254：kBluetoothAudioDevicePropertyFormat request 0 -> 1
09:14:36.255：声音核心执行配置切换，选择 16 kHz、单声道通话端点
09:14:36.287：Current profile tsco
```

其他关键证据：

- 09:14:36.196：`audioaccessoryd` 决定把 Redmi 设为临时默认输入；
- 09:14:36.200：Redmi 成为临时默认输入；
- 本窗口没有微信输入法的 `StartIO`；
- 格式请求进程可反查为微信输入法 `com.tencent.inputmethod.wetype`；
- 重新启动微信输入法并保持空闲时，约 40 次读取保持 `44.1 kHz / 双声道 / A2DP`，没有新的格式请求。

上游触发线索但尚未确证：

- 用户通过组合键启动微信语音输入，并将它绑定到会自动休眠的外置蓝牙小键盘；
- 小键盘唤醒后可能需要等待重连，用户在语音面板未及时出现时会重复按键；
- 异常期间微信输入法菜单项可能减少，打开应用或设置页后暂时恢复；
- 本机存在把外设发出的“左 Option + A”转换为微信语音快捷键“右 Option + 空格”的规则；
- 没有对应时刻的原始按键事件，因此不能把外置键盘或重复按键确认为根因。

退出 HFP 的方法：退出微信输入法，确认它没有自动重启，然后断开并重新连接设备。

恢复结果：2026-07-18 09:22:32 起连续六次读取为输入 `16 kHz / 单声道`、输出 `44.1 kHz / 双声道`，日志持续显示 `Current profile tacl`。

结论边界：已确认微信输入法是这次事件的直接格式请求者；微信输入法为什么产生这条请求仍待确证。不能写成“微信输入法只要运行就必然触发 HFP”。

## C4：蓝牙输入 A 导致蓝牙输出 B 进入 HFP

宿主机：MAC-home02-MacMini；Mac mini（M1）；机型标识 `Macmini9,1`；硬件唯一标识 `175AC621-14A2-5E02-AEFD-F7D360137359`；macOS 15.7.7 / 24G720；蓝牙控制器 `BCM_4378`

输入设备：

- 名称：`Redmi电脑音箱-3899`
- 唯一标识：蓝牙地址 `50:88:11:07:63:DA`
- 角色：蓝牙输入 A
- 状态：默认输入；实验中没有应用实际采集输入

输出设备：

- 名称：`REDMI Buds 6 Pro 电竞版`
- 唯一标识：蓝牙地址 `00:BB:43:41:6B:81`
- 角色：蓝牙输出 B
- 状态：默认输出

问题现象：选择蓝牙输入 A 后，另一台蓝牙输出 B 从 `tacl`、约 44.1 kHz、双声道切换为 `tsco`、16 kHz、单声道。

原因：在该主机、系统版本和设备组合中，macOS 原生蓝牙声音路由因“蓝牙输入 A + 蓝牙输出 B”组合而联动降级输出 B。

关键日志原文：

```text

08:36:04.576：macOS 原生蓝牙声音服务发送设备配置变化
08:36:04.661：系统为耳机输出选择 16 kHz，立体声关闭
08:36:04.728：耳机输出变为 Current profile tsco
08:36:05.805：系统记录 1 个 SCO 设备、0 个 A2DP 设备
```

持续性证据：

- 关闭控制中心和声音设置页面后，HFP 仍持续到至少 08:39:42；
- 没有观察到 A 的输入端开始采集；
- 蓝牙链路日志持续显示 `AudioInput: 0`；
- SoundSource 已退出并停止应用级声音路由；
- 切回非蓝牙输入后，B 断开 `tsco` 并恢复 `tacl`、44.1 kHz、双声道。

退出 HFP 的方法：把默认输入切回 USB、内置麦克风或其他非蓝牙输入，等待 B 恢复；必要时重新选择输出设备，让应用重建声音队列。

恢复结果：切回 `MOONDROP Rays` 后，B 恢复 `tacl`、约 44.1 kHz、双声道，并重新出现 1 个 A2DP、0 个 SCO。

其他恢复实测：

- 在目标设备仍同时作为默认蓝牙输入和输出时，重新提交相同输出路由，仍为 16 kHz、单声道，失败；
- 完整断开并重连目标设备，重新设为默认输出，仍为 16 kHz，失败；
- 使用 USB 输入替代蓝牙输入是优先候选，但“USB 输入实际采集 + B 持续播放”的完整验收尚未完成；
- 禁用 B 的声音输入功能、使用专用 USB/2.4G 接收器、组合声音设备或格式锁定，均未形成已验证的恢复方法。

结论边界：这是特定主机、系统、输入设备 A 和输出设备 B 的组合事实，不能推广为“默认输入本身等于 HFP”，也不能推广到所有蓝牙设备组合。

## C7：微信输入法把两个不同蓝牙端点放进同一声音会话后进入 HFP

- 唯一标识：蓝牙地址 `50:C0:F0:F3:6A:66`

问题现象：微信输入法语音输入短暂出现波形，随后耳机反复提示“已断开、已连接”，输入没有声音，XIBERIA 输出进入 HFP/HSP。

原因：微信输入法把一个蓝牙输出端点和另一台蓝牙设备的输入端点同时放进同一个声音会话，macOS 拒绝这组双蓝牙端点配置，随后格式请求反复切换。

```text
  input_running: true
  output_running: false

There was an error setting the deviceUUIDs as there are more than one BT device connected = (
    "50:C0:F0:F3:6A:66",
    "58:B8:58:9D:C1:E8"
)
```

其他关键事实：

- 失败窗口约为 2026-07-18 15:28:17 至 15:28:22；
- XIBERIA 输出在 16 kHz 和 44.1 kHz 之间反复切换；
- 微信输入法使用 DJI 输入、MacBook 内置扬声器输出时成功；
- 微信应用内置语音转文字使用 DJI 输入、BOSE 输出时成功，因为实际会话只带 DJI 输入；
- 微信输入法输入输出都使用 XIBERIA 时成功。

恢复结果：切换为非蓝牙输出或同一蓝牙设备输入输出时，实例对照成功；原失败组合的完整自动恢复结果待确证。

结论边界：不能扩大为“微信输入法只要使用蓝牙麦克风就必然失败”，也不能只看系统默认输入输出，必须检查实际声音会话里的端点列表。

## C8：Bose 输出被系统联动为输入后进入 HFP

宿主机：`MAC-home01-MacBookAir`；`andymacbook-air.local`；macOS 14.6.1

输入设备：

- 名称：Bose QC Headphones
- 唯一标识：本实例摘要未留存蓝牙地址
- 状态：选择 Bose 输出后被系统联动为输入

输出设备：

- 名称：Bose QC Headphones
- 唯一标识：本实例摘要未留存蓝牙地址
- 状态：默认输出

问题现象：Bose 从 `44.1 kHz / 双声道` 切为 `16 kHz / 单声道`，传统蓝牙通话链路建立；语音结束后 QQMusic 出现卡顿、静音和恢复异常。

原因：系统把 Bose 输出和输入联动到同一双向通话链路。

关键日志原文：

```text
Starting IO on profile tsco
最佳采样率 16000
stereo 0
可用 CVSD 和 mSBC
```

退出 HFP 的方法：将输入切换到 Redmi、Mac 内置麦克风或 USB 麦克风，同时保留 Bose 为输出。

恢复结果：输入切到 Redmi 后，Bose 仍作为输出，QQMusic 可以继续播放，用户确认听感正常。

结论边界：不能简化成“Bose 只要被设为默认输入就必然进入 HFP”，必须结合实际采样率、声道和链路日志判断。

## C9：K03S 蓝牙输入输出切换后进入 HFP

宿主机：`MAC-home01-MacBookAir`；`andymacbook-air.local`；macOS 14.6.1

输入设备：

- 名称：XIBERIA K03S
- 唯一标识：本实例摘要未留存蓝牙地址
- 状态：输入输出都使用 K03S 的蓝牙端点

输出设备：

- 名称：XIBERIA K03S
- 唯一标识：本实例摘要未留存蓝牙地址
- 状态：默认输出

问题现象：K03S 从 `44.1 kHz / 双声道` 降为 `16 kHz / 单声道`；输入切走后可能短暂恢复又再次降级，QQMusic 出现输出队列停止、启动和控制延迟。

原因：蓝牙输入输出切换导致设备格式重新协商，应用输出队列重建与设备恢复不同步。

关键日志原文：

```text
tacl
tsco
QQMusic 输出队列停止、启动、再次停止、启动
```

退出 HFP 的方法：将 K03S 改用 2.4G 接收器，使系统把它识别为 USB 声音设备；或将输入切换到 Mac 内置、USB 或其他非目标蓝牙设备。

恢复结果：2.4G 接收器模式下，监视器记录约 `48 kHz` 输出双声道、`48 kHz` 输入单声道，用户确认输入输出切换不再出现本实例异常。

对照细节：同一副 K03S 使用 2.4G 接收器时，K03S 输入、Bose 输出以及多次输入输出切换均无异常；切回传统蓝牙后，输入输出都为 K03S 时出现 16 kHz 单声道，输入切走后可能长时间无声或恢复异常，切换过快时 QQMusic 暂停控制延迟。

结论边界：只能说明本机本次实验中更换底层链路后异常消失，不能把每次无声都归因于同一个内部软件缺陷，也不能把 2.4G 接收器写成 LE Audio（低功耗蓝牙音频）证明。

## 统一排除项

以下现象不能单独作为进入 HFP 的原因：

- 蓝牙设备只是被设为系统级或应用级默认输入；
- 设备显示 16 kHz、单声道或 `tsco`；
- HFP 服务晚于 A2DP 服务接入；
- `audioaccessoryd` 设置临时默认输入；
- 重启声音核心进程后重新出现 HFP；
- 音量异常或独立音量端点变化。

这些只能作为状态、系统动作、复现条件或伴随现象，必须继续寻找实际麦克风读取、格式请求、输入输出联动或其他完整证据链。

## 统一恢复记录规则

每个实例都必须单独记录退出 HFP 的方法和结果。只有同时满足以下条件，才能写“恢复成功”：

1. 目标输出仍是目标设备；
2. 实际输出连续六次高于 16 kHz；
3. 相邻读取至少间隔 500 ms；
4. 记录了恢复前后输入输出设备和关键日志；
5. 若涉及进程退出，记录进程是否真正退出以及是否仍在读取麦克风。

没有满足条件时，写“待确证”或“观察到恢复但未达到验证标准”。

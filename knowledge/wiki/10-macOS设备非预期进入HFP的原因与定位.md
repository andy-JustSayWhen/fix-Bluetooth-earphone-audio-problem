# macOS 设备非预期进入 HFP 的原因与定位

## 适用范围

本文汇总本项目在多个任务、本机日志和重复对照实验中观察到的 HFP 触发原因。实测结论默认绑定到 `MAC-home02-MacMini`：Mac mini（M1）、macOS 15.7.7（系统内部版本 `24G720`）。其他电脑、系统版本和蓝牙设备组合必须重新验证。

## 统一定位顺序

1. 确认目标设备确实是默认输出，并记录当前采样率、声道、`tacl/tsco` 和 A2DP/SCO 数量。
2. 以首次 `tacl -> tsco` 或高采样率降到 16 kHz 的时间为中心，截取前后至少 2 秒日志。
3. 查 `StartIO`、录音状态和麦克风权限请求，区分“真正录音”与“仅探测或配置”。
4. 查蓝牙格式请求中的进程编号，并用进程表映射到具体程序。
5. 查 `audioaccessoryd` 是否把蓝牙输入设为临时默认输入；这属于系统执行动作，不能自动当作最终根因。
6. 退出候选程序后重复完全相同的断开重连或输入切换，并保持最终输入、输出条件不变。
7. 连续六次确认输出高于 16 kHz、双声道且仍为目标默认输出，才把候选原因升级为确证。

## 不能单独当作根因的现象

| 现象 | 原因 |
| --- | --- |
| 设备显示 16 kHz、单声道或当前为 `tsco` | 只能证明已经进入 HFP，不能说明谁触发 |
| HFP 服务晚于 A2DP 服务接入 | 只是连接顺序；仍需查后续格式请求或输入路由动作 |
| `audioaccessoryd` 设置临时默认输入 | 是系统执行动作；需要继续追查上游应用或系统策略 |
| 重启声音核心进程后进入 HFP | 是复现条件；可能重新触发默认输入恢复和应用探测 |
| 音量异常 | 可能是 HFP 独立音量端点的伴随现象，不能证明触发来源 |

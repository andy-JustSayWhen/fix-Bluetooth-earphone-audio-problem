# 选择蓝牙 A 作为输入导致蓝牙 B 进入 HFP

## 已验证的实验环境

本文的实测结论绑定到以下环境，环境记录号为 `MACMINI9,1-175AC621`：

| 类型 | 名称 | 唯一标识或固定版本 |
| --- | --- | --- |
| 实验主机 | Mac mini，Apple M1 | 机型标识 `Macmini9,1`；硬件 UUID（像身份证号一样识别硬件的编号）`175AC621-14A2-5E02-AEFD-F7D360137359`；序列号 `C07FG7XUQ6NV` |
| 操作系统 | macOS | 15.7.7；系统内部版本 `24G720` |
| 蓝牙控制器 | 主机内置控制器 | 芯片 `BCM_4378` |
| 蓝牙输入 A | `Redmi电脑音箱-3899` | 蓝牙地址 `50:88:11:07:63:DA` |
| 蓝牙输出 B | `REDMI Buds 6 Pro 电竞版` | 蓝牙地址 `00:BB:43:41:6B:81` |
| 非蓝牙输入对照 | `MOONDROP Rays` | USB（用线连接电脑的接口）序列号 `00000000000829`；厂商编号 `0x35d8`；产品编号 `0x011b` |

设备名称可以被用户修改，不能单独充当身份依据。复测或合并新案例时，应优先比较上述硬件唯一标识、蓝牙地址和系统内部版本。

## 结论

统一口径：系统级或应用级默认输入本身不等于 HFP。本文记录的是一个已经被采样率、声道、`tacl/tsco` 链路和设备地址共同确认的特定跨设备异常；不能把它推广为所有蓝牙默认输入都会触发 HFP。

在实验环境 `MACMINI9,1-175AC621` 上，只要把 `Redmi电脑音箱-3899`（蓝牙地址 `50:88:11:07:63:DA`）设为默认输入，正在承担输出的 `REDMI Buds 6 Pro 电竞版`（蓝牙地址 `00:BB:43:41:6B:81`）就会被系统从 `tacl`（蓝牙立体声传输链路）切换到 `tsco`（蓝牙通话传输链路）。耳机输出随之从约 44.1 kHz、双声道降为 16 kHz、单声道，即使：

- A 与 B 是两个不同的蓝牙设备；
- B 的麦克风没有被选中；
- A 的输入端没有被应用实际采集；
- 控制中心和系统声音设置页面已经关闭；
- SoundSource 已退出并停止应用级声音路由。

因此，这个现象不能仅用“同一副蓝牙耳机同时收音和播放时进入 HFP（蓝牙通话模式）”解释。当前证据指向 macOS 原生蓝牙声音路由在“蓝牙输入 A + 蓝牙输出 B”组合下实施了跨设备通话模式联动。

本结论只适用于上表明确记录的主机、系统和声音设备组合，不能直接推广为其他 macOS 版本、蓝牙芯片或设备组合的普遍行为。

## 与正常 HFP 行为的区别

传统蓝牙耳机使用自身麦克风时，需要双向传输，系统把同一设备的输出降为单声道通话链路，是已有官方资料与本项目实测共同支持的正常机制。参见[传统蓝牙麦克风导致播放质量下降](01-传统蓝牙麦克风导致播放质量下降.md)。

本主题记录的是不同组合：

```text
默认输入：蓝牙设备 A
默认输出：蓝牙设备 B

选择 A 输入
  -> macOS 原生声音服务要求 B 改变传输配置
  -> B：tacl / 44.1 kHz / 双声道
  -> B：tsco / 16 kHz / 单声道
```

现有 Apple 支持文档解释的是“应用使用蓝牙耳机自身麦克风后，该耳机播放质量降低”，没有把“选择另一台蓝牙设备作为输入，会让当前输出耳机进入 HFP”描述为标准行为。因此本文把跨设备联动记录为环境 `MACMINI9,1-175AC621` 的实测事实和 macOS 实现层现象，不写成 HFP 规范要求，也不擅自断言 Apple 已正式认定为软件缺陷。

## 本项目的判定证据

可靠判定不能只看设置界面中的设备名称，应同时满足以下证据链：

1. 默认输入的设备地址属于 A，默认输出的设备地址属于 B，证明两者不是同一台设备。
2. 切换前，B 的系统日志明确显示 `Current profile tacl`，输出为约 44.1 kHz、双声道。
3. 选择 A 作为输入后，B 收到传输配置变化请求。
4. B 的最佳采样率变为 16 kHz，立体声标记关闭，随后日志显示 `Current profile tsco`。
5. 系统无线共存状态从一个 A2DP（蓝牙高质量立体声播放模式）设备变为一个 SCO（蓝牙通话声音传输）设备。
6. 切换回 USB 等非蓝牙输入后，B 的 `tsco` 断开并恢复 `tacl`、44.1 kHz、双声道。

只有界面选中状态或只有 16 kHz、单声道，都不足以单独证明完整因果链。

## 已排除的替代解释

### 不是 SoundSource 应用级路由

环境 `MACMINI9,1-175AC621` 的对照中，SoundSource 退出后逐一撤销应用静音并停用虚拟声音设备。它的后台代理和驱动仍保持加载，但后续没有再建立虚拟路由。跨设备切换由控制中心提交给 macOS，配置变化由 macOS 原生声音服务执行，现象仍能稳定复现。

因此，“SoundSource 主程序退出但驱动仍加载”不能单独算作完全卸载；不过本次日志已经能区分“组件驻留”与“实际参与路由”，并确认后者没有发生。

### 不是声音设置页面持续打开

用户选择蓝牙设备 A 作为输入后立即关闭控制中心和声音设置页面。B 仍在约 0.54 秒内切到 16 kHz、单声道，并持续保持通话链路超过一分钟，没有自动恢复。

### 不是后台应用正在录音

复现窗口内没有观察到 A 的输入端开始采集。蓝牙链路质量日志持续显示 `AudioInput: 0`，相关输入会话处于停止状态，但 B 仍保持 `tsco`。这说明在当前复现路径中，设置 A 为默认蓝牙输入本身就足以触发 B 降级，不需要应用真正读取麦克风数据。

### 不是耳机 B 自行切换

日志顺序是：控制中心设置 A 为默认输入，macOS 原生声音服务向 B 发出配置变化，B 随后从 `tacl` 变成 `tsco`。因此不能把行为描述为 B 在没有系统请求的情况下自行误判。

## 影响

当蓝牙输出 B 被跨设备联动到 HFP 后，可能出现：

- 音乐由双声道降为单声道，听感空洞、低码率或失真；
- 应用在设备重配置期间短暂无声；
- 播放任务仍显示进行，但声音队列正在等待设备恢复；
- 输入切回非蓝牙设备后，底层链路已恢复，应用仍可能需要重新建立输出。

“没有声音”不代表输出设备一定选错；默认输出仍可以是 B，只是 B 当前处在不适合普通音乐播放的通话链路。

## 当前可用的规避方式

如果目标是让蓝牙耳机 B 保持正常立体声输出，环境 `MACMINI9,1-175AC621` 当前最稳定的做法是：

1. 默认输入使用 USB、内置麦克风或其他非蓝牙输入。
2. 默认输出保持蓝牙耳机 B。
3. 若 B 已进入 `tsco`，先把输入切回非蓝牙设备，等待 B 恢复 44.1/48 kHz、双声道。
4. 若应用仍无声，再暂停并重新播放，或重新选择一次输出设备，让应用重建声音队列。

这是一种绕过 macOS 跨设备蓝牙路由行为的方案，不是修改 HFP 能力上限。

完整的软件、组合设备和硬件方案分级见[跨设备蓝牙 HFP 降级的解决方案边界](07-跨设备蓝牙HFP降级的解决方案边界.md)。当前优先候选是让 `Redmi 电脑音箱` 使用其 USB 输入端点，但该候选还需要在真实录音与 REDMI Buds 持续播放同时进行的条件下验收。

## 如何复测

建议使用固定顺序，并在每一步保留系统日志：

```text
非蓝牙输入 + 蓝牙 B 输出
  -> 确认 B 为 tacl / 双声道
  -> 输入切到蓝牙 A
  -> 立即关闭控制中心和声音设置
  -> 等待至少 10 秒
  -> 检查 B 是否变为 tsco / 16 kHz / 单声道
  -> 输入切回非蓝牙设备
  -> 检查 B 是否恢复 tacl / 双声道
```

更换 macOS 版本、Mac、蓝牙适配器、设备 A 或设备 B 后，都应作为新设备组合重新验证，不能沿用本案例结论。

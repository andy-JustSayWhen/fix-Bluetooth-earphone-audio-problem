# 防止设备进入 HFP 的方法

## 文档定位

本文档持续收录“在使用前避免目标蓝牙音频设备进入 HFP”的方法。这些方法改变使用条件或应用路由，不等于修复已经发生的链路残留，也不等于修改蓝牙协议能力。

每个方法必须分别写清已确认条件、可复现程度、失效条件和验收依据。单机单次成功只能标为候选绕过，不得写成通用规则。

## 方法一：让播放应用单独指定非默认蓝牙设备为输出

### 操作条件

1. 目标蓝牙音频设备已连接，但不是系统默认输入，也不是系统默认输出。
2. 系统默认输入和输出改用其他设备。本轮用户观察中，其他设备可以是蓝牙、有线或内置设备；当前尚未完成逐组合复测。
3. 播放应用本身支持单独选择输出设备。
4. 在应用内把目标蓝牙设备选为当前输出，但不改变系统默认输出。
5. 播放期间不让该应用或其他进程读取目标设备的麦克风。

### 本次为什么有效

2026-07-21 的 QQ音乐 + K03S 实测中，系统日志显示：

- QQ音乐会话为媒体播放，标记为正在播放、没有录音。
- 会话设备列表只有 K03S 的输出端点，没有 K03S 或系统默认麦克风的输入端点。
- 系统为 K03S 启动 `tacl` 和 A2DP 播放流，端点保持 `44.1 kHz / 2 声道`。

这组证据表明，本次成功的直接链路是“应用对目标设备建立了输出专用播放会话，系统选择了 A2DP”。“目标设备不是系统默认设备”是本方法的操作条件，但尚未证明它单独就是避免 HFP 的充分条件。

### 对照结果

同一轮实验中，当 K03S 被改成系统默认输出、DJI Mic Mini 仍为系统默认输入时，K03S 曾从 `tacl` 进入 `tsco`。将系统默认输出移走后，系统关闭 K03S 的 HFP 流并恢复 A2DP，QQ音乐继续直接向 K03S 播放。

这一对照支持把方法一作为候选绕过，但仍不足以证明所有 macOS 版本、所有应用或所有蓝牙设备都会遵循同样的选路。

### 验收方法

不能只听到声音就认为本方法生效。每次必须同时核对：

1. 目标设备仍然不是系统默认输入或默认输出。
2. 播放应用的会话只声明目标输出端点，`input_running` 为否。
3. 目标设备最新声音链路为 `tacl`，并且系统记录 A2DP 流启动。
4. 目标输出的实际采样率高于 `16 kHz`、声道数不少于 2，连续三次复核保持一致，每次间隔 `500 ms`。
5. 没有进程正在读取目标设备的实体麦克风端点。

### 限制与失效条件

- QQ音乐界面已注明“修改仅在此次运行有效”，下次启动不能假定仍保留该设备选择。
- 当前只有 `andymacbook-air` + QQ音乐 9.2.5 + XIBERIA K03S 的一轮实测。
- 未验证重启应用、重启系统、更换 Mac、更换蓝牙设备、更换播放应用或长时间播放。
- 如果任何进程读取目标设备的麦克风，或应用会话同时声明蓝牙输入端点，目标仍可能进入 HFP。
- 本方法只改变特定应用的输出；系统声音和其他应用仍走系统默认输出。
- 已经进入 HFP 且存在链路残留时，本方法只能作为促使系统重新评估播放链路的候选动作，不能替代一键修复路由。

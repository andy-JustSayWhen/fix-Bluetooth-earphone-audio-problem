# 2026-07-20 格式请求类前端实测

返回：[HFP 一键修复前端实测方法](../00-HFP一键修复前端实测方法.md)

## 测试例

进入格式请求类必须同时满足：

1. 没有进程实际读取目标蓝牙麦克风。
2. 普通进程提交 `kBluetoothAudioDevicePropertyFormat request 0 -> 1`。
3. 两秒内出现 `Current profile tsco`。
4. 同一进程在同一窗口没有 `BluetoothHALPlugIn_StartIO`。
5. 只有一个低采样率蓝牙输出目标，且当前进程启动时间没有晚于日志事件。

## 本轮实测事实

- 第一次微信输入法调用实际读取了 DJI 麦克风，前端直接识别为 `WeType` 占用，不满足“没有实际麦克风占用”。
- 结束 `WeType` 后，重新启动的进程没有恢复免提语音快捷键；因此本轮无法再次产生格式请求，不形成新的 C3 样本。
- 实时日志在后续尝试中只记录到 K03S 输出侧的 `aceagent` 与系统提示音会话，没有微信输入法格式请求；不能据此认定格式请求类发生。

## 实测发现并修复的基础问题

一键修复在二次原因判定时曾把标准时间字符串直接传给系统日志命令，真实日志报：

```text
Failed conversion ... using format '%Y-%m-%d %H:%M:%S'
```

该问题会使多端点和格式请求诊断读取不到日志。实现现已改为本地 `年-月-日 时:分:秒` 格式；修复后现场查询返回 `queryError: null`。相关自动测试已加入。

## 结论边界与下次步骤

- 本轮没有重新复现格式请求类，不能把历史 C3 证据写成本轮通过。
- 历史已确证样本见 [蓝牙音频设备进入HFP模式的原因](../../蓝牙音频设备进入HFP模式的原因.md) 的 C3。
- 下次应在不结束微信输入法进程的独立测试会话中执行：先恢复 A2DP，启动实时日志，调用一次免提语音；若先出现实际占用，立即归占用类并停止格式请求判定。

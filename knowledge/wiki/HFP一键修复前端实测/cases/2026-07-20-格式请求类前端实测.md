# 2026-07-20 格式请求类前端实测

返回：[HFP 一键修复前端实测方法](../00-HFP一键修复前端实测方法.md)

## 测试例

进入格式请求类必须同时满足：

1. 没有进程实际读取目标蓝牙麦克风。
2. 普通进程提交 `kBluetoothAudioDevicePropertyFormat request 0 -> 1`。
3. 两秒内出现 `Current profile tsco`。
4. 同一进程在同一窗口没有 `BluetoothHALPlugIn_StartIO`。
5. 只有一个低采样率蓝牙输出目标，且当前进程启动时间没有晚于日志事件。

## 本轮实测事实

- 2026-07-20 09:54:12，微信输入法 `WeType` 进程号为 40496，启动时间为当日 02:54，排除了进程号复用。
- 09:54:12.181，系统记录 `kBluetoothAudioDevicePropertyFormat request 0 -> 1`；09:54:12.270 记录 `Current profile tsco`。
- 同一进程两秒内没有 `BluetoothHALPlugIn_StartIO`，现场也没有实际麦克风占用。K03S 同期变为 16 kHz、单声道，因此该次完整命中格式请求类。
- 09:59:27 首次通过前端执行一键修复，K03S 恢复到 44.1 kHz。但当时原因分类被错误降为证据不足：程序把非默认 DJI 设备的上次 16 kHz 活动值也当成了当前低采样率输出。
- 实现已改为只统计当前默认蓝牙输出，并增加回归测试。由于本轮没有第二次真实触发，修正后的“前端点击直接命中格式请求类”仍需下次实机补验。

## 实测发现并修复的基础问题

一键修复在二次原因判定时曾把标准时间字符串直接传给系统日志命令，真实日志报：

```text
Failed conversion ... using format '%Y-%m-%d %H:%M:%S'
```

该问题会使多端点和格式请求诊断读取不到日志。实现现已改为本地 `年-月-日 时:分:秒` 格式；修复后现场查询返回 `queryError: null`。相关自动测试已加入。

## 结论边界与下次步骤

- 本轮通过：新的真实格式请求证据链、前端修复后恢复 44.1 kHz。
- 本轮未通过也未判失败：分类统计错误修正后，没有新的真实触发可供第二次前端验证。
- 下次应先确认微信输入法语音快捷键可用，恢复 A2DP，启动实时日志，调用一次免提语音；若先出现实际占用，立即归占用类并停止格式请求判定。

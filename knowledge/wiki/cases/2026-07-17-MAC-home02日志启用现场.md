## 与 HFP 现场的关系

核查时统一日志仍每秒出现 `HFP LinkQualityReport`，设备名称为 `Redmi电脑音箱-3899`。这证明蓝牙详细日志配置已经生效，也证明当时通话链路仍存在；但日志能力是否启用本身不能恢复 A2DP。

## 保持默认输入输出不变的重启实测

### 重启前基线

- 默认输入：`Redmi电脑音箱-3899`
- 默认输出：`Redmi电脑音箱-3899`
- 输入：`16 kHz / 单声道`
- 输出：`16 kHz / 单声道`
- 页面接口判定：`HFP_HSP`

### 重启后证据

- 自动采集于 `2026-07-17 22:14:45 +08:00` 开始。
- `22:14:49.046`：蓝牙基础连接成功。
- `22:14:50.401`：系统创建该设备的 HFP 声音端点，并选择 `mSBC` 通话编码。
- `22:14:51.092`：系统发布 A2DP 声音端点。
- `22:14:51.097`：`coreaudiod` 建立 `tacl` 端点连接。
- `22:14:51.149`：系统明确记录当前输出配置为 `tacl`。
- 随后的链路报告持续显示 `A2DP_CONNECTION_COUNT = 1`、`SCO_CONNECTION_COUNT = 0`，并每秒产生 `A2DP LinkQualityReport`。
- 从自动采集的第一份声音快照开始，默认输入仍为同一音箱的 `16 kHz / 单声道`，默认输出已经是 `44.1 kHz / 双声道`。

### 已确证结论

本次重启没有改变用户选择的默认输入、输出设备，但重建了蓝牙声音端点，并使活动输出从重启前的 HFP 通话输出变为 A2DP 高音质输出。重启后用户实际播放并确认听感正常，因此系统参数恢复与主观听感恢复已经相互验证。

## 重启后 Codex 语音自动恢复实测

用户在上述重启恢复后再次启用 Codex 语音，输入、输出仍保持为 `Redmi电脑音箱-3899`。本次日志完整记录了进入和退出通话链路的过程：

- `22:18:48.718`：`Codex (Service)` 向蓝牙声音插件注册。
- `22:18:48.947`：Codex 开始读取 `tsco` 输入端点。
- `22:18:48.968`：当前输出配置由 `tacl` 切换为 `tsco`，即从高音质输出进入通话输出。
- `22:18:56.643`：语音结束后，系统开始断开该设备的 HFP 声音连接。
- `22:18:56.783`：系统明确记录正在断开 SCO。
- `22:18:56.800`：`tsco` 传输断开完成。
- `22:18:56.806`：当前输出配置重新切换为 `tacl`。
- `22:18:56.909`：A2DP 声音流重新开始。
- 此后持续记录 `A2DP_CONNECTION_COUNT = 1`、`SCO_CONNECTION_COUNT = 0`，当前输出为 `44.1 kHz / 双声道`。
- 用户确认语音结束后的实际听感正常。

本轮自动恢复的直接原因已经确证：Codex 结束麦克风读取后，系统成功释放 SCO 通话传输，并重新激活 `tacl` 高音质输出端点。重启后 SoundSource 和 `arkaudiod` 均未运行，`ARK.driver` 仍位于停用目录；这是与此前无法自动恢复时不同的环境条件，但本次单次对照不足以把 ARK 单独确认为此前卡住的唯一原因。

### 后续恢复 SoundSource 环境

为准备后续对照复现，已将 `ARK.driver` 从停用目录移回 `/Library/Audio/Plug-Ins/HAL/ARK.driver`，重启系统声音服务并启动 SoundSource。恢复后核验结果：

- `ARK.driver` 位于正式声音驱动目录，代码签名验证通过。
- SoundSource 前台进程正在运行。
- `arkaudiod --agent` 后台进程正在运行，对应系统任务状态为 `running`。
- 恢复组件后，`Redmi电脑音箱-3899` 仍保持默认输入 `16 kHz / 单声道`、默认输出 `44.1 kHz / 双声道`。

因此，“恢复 SoundSource 就立即进入 HFP”不成立。后续复现需要再次触发真实麦克风读取，并观察语音结束后 SCO 是否未被释放。

# 微信输入法调用蓝牙麦克风失败：跨蓝牙输入输出会话对照

## 文档性质

本文是本机一次完整排障过程的故障实例记录，不是对所有 macOS 电脑、所有蓝牙设备或所有版本微信输入法的普遍结论。

本文只记录系统日志和本地检查器日志能够直接支持的事实。用户后续正常使用微信输入法语音功能的样本属于正常行为，不能仅因为它触发了蓝牙麦克风就把它标记为异常。

## 本机与设备身份

| 项目 | 已确认值 |
| --- | --- |
| 主机名 | `andymacbook-air` |
| 机型 | `MacBookAir10,1`，Apple M1 |
| BOSE | `Bose QC Headphones`，蓝牙地址 `E4:58:BC:3D:00:E7` |
| DJI | `DJI Mic Mini-9DC1E8`，蓝牙地址 `58:B8:58:9D:C1:E8` |
| XIBERIA | `XIBERIA K03S`，蓝牙地址 `50:C0:F0:F3:6A:66` |
| 微信输入法进程 | `WeType`，程序标识 `com.tencent.inputmethod.wetype` |
| 微信应用进程 | `WeChat`，程序标识 `com.tencent.xinWeChat` |

本案例与另一台 Mac mini 上的 Redmi 案例不是同一台电脑、也不是同一组设备，不能混合引用。本文只分析上表这台 MacBook Air 的 BOSE、DJI 和 XIBERIA。

## 用户可见现象

在本机把一个蓝牙设备作为输入、另一个蓝牙设备作为输出时，微信输入法语音输入可能短暂出现波形，随后耳机反复提示“已断开、已连接”，输入没有声音，目标蓝牙输出进入 HFP/HSP（蓝牙通话模式）。

对照实验显示：

1. 微信输入法使用 XIBERIA 输入、XIBERIA 输出时，可以持续使用。
2. 微信输入法使用 DJI 输入、MacBook 内置扬声器输出时，可以持续使用。
3. 微信应用内置语音转文字使用 DJI 输入、BOSE 输出时，也可以成功；这一次系统桌面看到的输出虽为 BOSE，但微信应用的录音会话没有把 BOSE 输出端点带入会话。
4. 微信输入法使用 DJI 输入、XIBERIA 输出时失败，并出现系统明确的双蓝牙设备错误。

## 失败样本：微信输入法把两个蓝牙端点放进同一个声音会话

### 失败现场

时间窗口：2026-07-18 15:28:17 至 15:28:22 左右。

用户当时的路由为：

| 方向 | 设备 |
| --- | --- |
| 输入 | DJI Mic Mini-9DC1E8 |
| 输出 | XIBERIA K03S |

系统声音日志中的关键记录如下：

```text
15:28:17.939653  session: WeType(61338)
details:
  deviceUIDs:
    - 50-C0-F0-F3-6A-66:output
    - 58-B8-58-9D-C1-E8:input
  implicit_category: Record
  input_running: true
  output_running: false
```

紧接着系统记录：

```text
There was an error setting the deviceUUIDs as there are more than one BT device connected = (
    "50:C0:F0:F3:6A:66",
    "58:B8:58:9D:C1:E8"
)
```

随后，微信输入法进程 `WeType`（进程编号 `61338`）提交的蓝牙音频格式请求反复变化，系统观察到 XIBERIA 输出在 `16 kHz` 和 `44.1 kHz` 之间来回切换。这个切换对应用户听到的断开、重连和无声。

### 失败原因的直接解释

失败的直接原因不是“两个蓝牙设备同时连接”本身，而是：

> 微信输入法的录音会话同时携带了一个蓝牙输出端点和另一个蓝牙输入端点。

本次会话中的两个端点是：

```text
XIBERIA K03S:output
DJI Mic Mini-9DC1E8:input
```

系统在设置这组端点时明确报出 `more than one BT device connected`。之后格式请求不断重试，导致输出端在 A2DP（蓝牙高音质播放模式）和 HFP/HSP 之间抖动。

因此，本案例能确认的是“微信输入法提交的会话组合触发了 macOS 的双蓝牙端点冲突”，不能把原因扩大成“微信输入法只要使用蓝牙麦克风就必然失败”。

## 成功样本一：微信输入法使用 DJI 输入，内置扬声器输出

时间窗口：2026-07-18 15:29:32 至 15:30:00，以及 15:30:17 至 15:30:31。

用户路由为：

```text
输入：DJI Mic Mini-9DC1E8
输出：MacBook Air 内置扬声器
```

本次录音会话只出现 DJI 的输入端点，没有出现第二个蓝牙输出端点：

```text
deviceUIDs:
  - 58-B8-58-9D-C1-E8:input
```

微信输入法连续使用 DJI 麦克风约 27.8 秒，随后又使用约 14.6 秒，没有出现 `more than one BT device connected`，也没有出现 XIBERIA 那样的格式请求抖动。

这说明：同样是微信输入法、同样是蓝牙麦克风，输出改成非蓝牙设备后可以成功。

## 成功样本二：微信应用内置语音转文字使用 DJI 输入，BOSE 输出

时间窗口：2026-07-18 15:34:32 至 15:35:19。

用户路由为：

```text
系统默认输入：DJI Mic Mini-9DC1E8
系统默认输出：Bose QC Headphones
```

这一次是微信应用 `WeChat`（进程编号 `83476`）调用内置语音转文字，不是微信输入法 `WeType`。系统日志记录：

```text
15:34:32.086992  session: WeChat(83476)
details:
  deviceUIDs:
    - 58-B8-58-9D-C1-E8:input
  implicit_category: Record
  input_running: true
  output_running: false
```

微信应用随后以 `Record_WithBluetooth`（带蓝牙输入的录音会话）开始录音，并持续约 47.25 秒，最后正常释放。

关键差异在这里：虽然系统默认输出当时是 BOSE，但微信应用这次录音会话的 `deviceUIDs` 里只有 DJI 输入，没有：

```text
E4-58-BC-3D-00-E7:output
```

所以它没有把 BOSE 输出和 DJI 输入作为两个蓝牙端点一起交给同一个录音会话，因而没有触发本案例中的双蓝牙端点错误。

## 成功样本三：微信输入法输入输出使用同一个 XIBERIA

时间窗口：2026-07-18 15:26:22 至 15:27:15。

用户把输入和输出都切到 `XIBERIA K03S`。微信输入法进程 `WeType` 连续使用 XIBERIA 麦克风约 53 秒，并正常结束，设备随后恢复 A2DP。

这组样本说明：微信输入法并不是不能使用蓝牙麦克风；当输入输出属于同一个蓝牙设备时，本次测试没有出现两个不同蓝牙端点同时进入会话的冲突。

## 失败与成功的最简对照

| 场景 | 系统默认输入 | 系统默认输出 | 录音会话实际带入的蓝牙端点 | 结果 |
| --- | --- | --- | --- | --- |
| 微信输入法失败 | DJI | XIBERIA | `XIBERIA:output` + `DJI:input` | 报双蓝牙设备错误，格式反复切换 |
| 微信输入法成功 | DJI | MacBook 内置扬声器 | `DJI:input` | 成功 |
| 微信应用成功 | DJI | BOSE | 只有 `DJI:input` | 成功 |
| 微信输入法成功 | XIBERIA | XIBERIA | 同一蓝牙设备的会话 | 成功 |

## 结论

本机这次失败的关键不是“输入和输出都是蓝牙设备”这句话本身，而是微信输入法把两个不同蓝牙设备的端点同时放进了同一声音会话：`XIBERIA:output + DJI:input`。macOS 声音系统明确拒绝这组双蓝牙端点配置，随后产生格式请求抖动，最终表现为 HFP/HSP、断开重连和无声。

其他进程之所以成功，是因为它们的实际会话形状不同：

- 微信应用虽然桌面输出是 BOSE，但录音会话只声明 DJI 输入；
- 微信输入法把输出改为 MacBook 内置扬声器后，会话也只剩 DJI 输入；
- 输入输出同为 XIBERIA 时，没有两个不同蓝牙端点同时出现。

因此，前端或后续自动诊断不能只看“默认输入设备”和“默认输出设备”。必须继续读取实际声音会话里的端点列表，重点识别：

```text
不同蓝牙设备的 output + input 是否同时出现在同一会话中
```

## 结论边界

1. 当前直接证据证明的是微信输入法在本机一次失败会话中的端点组合和系统错误，不等于微信输入法所有版本、所有电脑都必然采用同样路径。
2. 当前日志证明了系统拒绝双蓝牙端点配置，但没有微信输入法源代码，不能声称已经知道它内部为什么把 XIBERIA 输出端点带入会话。
3. 不能把“BOSE 默认输出、DJI 默认输入”直接判为必然失败；本次微信应用内置语音转文字在同样的系统默认路由下成功，关键是它的实际会话只带 DJI 输入。
4. 这份记录用于“先定位原因，再快速一键恢复”的原因定位阶段。它还没有授权把结束微信输入法或切换系统输出直接写入一键恢复逻辑。

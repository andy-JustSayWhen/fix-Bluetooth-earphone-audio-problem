# 2026-07-17：REDMI 跨设备蓝牙输入输出联动（macOS 15.7.7 / 24G720）

## 案例归属

本案例是[选择蓝牙 A 作为输入导致蓝牙 B 进入 HFP](../06-选择蓝牙A输入导致蓝牙B进入HFP.md)在实验环境 `MACMINI9,1-175AC621` 上的实测记录。

## 设备与实验范围

本次实验发生在同一系统会话中。主机身份和系统版本如下：

| 项目 | 记录值 |
| --- | --- |
| 实验环境记录号 | `MACMINI9,1-175AC621` |
| 主机型号 | Mac mini，Apple M1 |
| 机型标识 | `Macmini9,1` |
| 硬件 UUID（像身份证号一样识别硬件的编号） | `175AC621-14A2-5E02-AEFD-F7D360137359` |
| 主机序列号 | `C07FG7XUQ6NV` |
| 操作系统 | macOS 15.7.7 |
| 系统内部版本 | `24G720` |
| 蓝牙控制器芯片 | `BCM_4378` |

参与实验的声音设备如下。名称只用于阅读，蓝牙地址或序列号才是区分设备的主要依据：

| 角色 | 设备名称 | 连接类型 | 唯一标识 |
| --- | --- | --- | --- |
| 蓝牙输出 B | `REDMI Buds 6 Pro 电竞版` | 蓝牙 | 蓝牙地址 `00:BB:43:41:6B:81` |
| 蓝牙输入 A | `Redmi电脑音箱-3899` | 蓝牙 | 蓝牙地址 `50:88:11:07:63:DA` |
| 非蓝牙输入对照 | `MOONDROP Rays` | USB（用线连接电脑的接口） | 序列号 `00000000000829`；厂商编号 `0x35d8`；产品编号 `0x011b` |

默认输出在整个关键对照中保持 `REDMI Buds 6 Pro 电竞版`。实验改变的是默认输入。

## SoundSource 排除过程

SoundSource 主程序退出后，系统日志在 08:35:13 至 08:35:22 记录其虚拟设备逐个停用，并撤销 QQ音乐、微信、浏览器、控制中心、系统声音设置等进程上的应用级静音。此后：

- `arkaudiod`（SoundSource 的后台声音处理进程）仍存在；
- ARK（SoundSource 使用的虚拟声音驱动）端点仍处于加载状态；
- 没有新的 `AddMuter`（给应用添加静音控制的动作）；
- 没有新的 ARK 虚拟声音设备被激活；
- 后续跨设备切换由控制中心、`coreaudiod`（macOS 管理声音设备的核心进程）和蓝牙声音服务完成。

因此本案例把“SoundSource 组件仍驻留”和“SoundSource 仍参与声音路由”分开判断；本次干净对照已排除后者。

## 第一轮干净切换时间线

### 输入切到 MOONDROP Rays

08:35:53.385，默认输入被设置为 MOONDROP Rays。REDMI 耳机输出最初仍保留上一状态的 `tsco`，随后发生恢复：

- 08:35:54.256：REDMI 耳机输出变为 `Current profile tacl`（当前使用蓝牙立体声传输链路）。
- 08:35:55.395：系统记录 0 个 SCO（蓝牙通话声音传输）设备、1 个 A2DP（蓝牙高质量立体声播放模式）设备。
- 设备参数恢复为约 44.1 kHz、双声道。

### 输入切到 Redmi电脑音箱-3899

08:36:04.120，默认输入被设置为地址 `50:88:11:07:63:DA` 的 Redmi电脑音箱。此时默认输出仍是地址 `00:BB:43:41:6B:81` 的 REDMI 耳机：

- 08:36:04.139：耳机输出仍为 `Current profile tacl`。
- 08:36:04.576：macOS 原生蓝牙声音服务发送设备配置变化。
- 08:36:04.661：系统为耳机输出选择 16 kHz，立体声关闭。
- 08:36:04.728：耳机输出变为 `Current profile tsco`（当前使用蓝牙通话传输链路）。
- 08:36:05.805：系统记录 1 个 SCO 设备、0 个 A2DP 设备。

这组顺序直接表明：选择蓝牙输入 A 后，被重新配置的是另一台蓝牙输出 B。

### 输入切到 REDMI 耳机自身麦克风

08:36:08.846，默认输入改为 REDMI 耳机自身输入；耳机输出继续保持 `tsco`。这是同一设备输入输出进入通话模式的正常对照，不是跨设备结论的唯一证据。

### 输入切回 MOONDROP Rays

08:36:12.126，默认输入再次改为 MOONDROP Rays：

- 08:36:12.886：REDMI 耳机的 `tsco` 完成断开。
- 08:36:13.032：耳机恢复 `tacl`。
- 08:36:12.970 附近的设备格式重建记录显示输出回到 44.1 kHz、双声道。
- 08:36:14.249：系统再次记录 0 个 SCO 设备、1 个 A2DP 设备。

## 关闭界面的持续性验证

08:38:38.241，用户再次把默认输入设置为 Redmi电脑音箱，并立即关闭控制中心和声音设置页面：

- 08:38:38.777：REDMI 耳机输出被设置为 16 kHz，立体声关闭。
- 08:38:40.090：系统记录 1 个 SCO 设备、0 个 A2DP 设备。
- 至少到 08:39:42，系统仍持续记录 1 个 SCO 设备、0 个 A2DP 设备。
- 实时设备查询仍显示 REDMI 耳机输出为 16 kHz、单声道。

因此，关闭声音设置界面不会解除该状态，排除了“页面为了显示输入音量而短暂占用麦克风”的充分解释。

## 输入未实际采集的验证

在上述持续窗口内：

- 没有观察到 Redmi电脑音箱输入端开始采集的记录；
- 输入法相关声音会话记录为停止；
- 蓝牙链路质量日志持续显示 `AudioInput: 0`（没有正在传送的蓝牙声音输入）；
- REDMI 耳机输出仍保持通话链路。

这支持本案例的关键判断：在 macOS 15.7.7（系统内部版本 `24G720`）及上述唯一标识对应的设备组合中，把蓝牙 A 设为默认输入本身就足以触发蓝牙 B 降级，不需要应用真正读取 A 的麦克风数据。

## 本案例结论

1. SoundSource 应用级路由不是环境 `MACMINI9,1-175AC621` 本轮跨设备联动的触发源。
2. 控制中心或声音设置页面保持打开不是维持通话链路的必要条件。
3. 后台应用实际采集蓝牙 A 的麦克风不是触发蓝牙 B 降级的必要条件。
4. macOS 原生声音服务明确向蓝牙 B 发起配置变化，并把 B 从立体声链路切到通话链路。
5. 切回非蓝牙输入后，B 能恢复立体声链路，说明 B 的硬件播放能力没有永久损坏。

本案例能证明硬件 UUID 为 `175AC621-14A2-5E02-AEFD-F7D360137359` 的 Mac mini，在 macOS 15.7.7（系统内部版本 `24G720`）及上述声音设备组合中存在跨设备蓝牙输入输出联动；它不能单独证明其他 Mac 或系统版本存在同一行为，也不能证明 Apple 内部把它归类为软件缺陷还是不受支持的设备组合限制。

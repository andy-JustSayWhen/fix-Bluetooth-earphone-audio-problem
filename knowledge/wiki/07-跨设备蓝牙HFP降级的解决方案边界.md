# 跨设备蓝牙 HFP（蓝牙通话模式）降级的解决方案边界

## 结论

对于“蓝牙输入 A 保持启用，同时要求原生蓝牙输出 B 永远保持 A2DP（蓝牙高质量立体声播放模式）”这一严格目标，目前没有找到 macOS 15.7.7 提供的、可验证且受支持的软件锁定开关。

当前方案应分成三类：

1. **高把握绕过**：让 A 或 B 至少一端以 USB（用线连接电脑的声音接口）或 2.4G（使用专用无线接收器的连接方式）接收器身份出现，使 macOS 不再同时管理两个传统蓝牙声音端点。
2. **改变输入选择**：自动把默认输入改为内置或 USB 麦克风，可以保护 B 的立体声输出，但不再满足“继续使用蓝牙 A 输入”。
3. **待验证的软件干预**：要求系统不要使用 B 的麦克风，理论上可能减少 B 进入 HFP（蓝牙通话模式）的机会；但本项目已经证明 B 麦克风未被选中时仍会发生跨设备降级，所以不能预先写成有效修复。

本主题讨论的是环境 `MACMINI9,1-175AC621`、macOS 15.7.7、系统内部版本 `24G720` 中发现的跨设备行为，不把其他平台或其他设备组合的结果直接套用。

## 为什么没有现成的协议锁定开关

Apple（苹果）的官方声音设置资料允许用户选择输入、输出、采样率，并通过 Audio MIDI Setup（macOS 自带的声音设备组合工具）把多个真实设备组合成一个逻辑设备。官方对组合设备的定义是增加输入输出通道、统一采样率、选择时钟源并进行时钟漂移修正，没有说明它能隔离蓝牙协议或强制某个子设备保持 A2DP。

因此，“组合设备可以阻止 B 进入 HFP”目前没有官方依据。组合设备仍引用底层真实蓝牙设备；是否会阻止本项目观察到的跨设备联动，必须实测，不能从“界面上变成一个新设备”推导出来。

来源：[Apple 组合声音设备原文](../raw/apple/apple-audio-midi-aggregate-device.md)。

Apple 官方关于蓝牙耳机音质下降的处理方式仍是停止使用耳机麦克风的应用、不要让系统设置停留在声音页面，必要时重新选择耳机。该页面没有提供“通话输入继续工作但输出强制保持高质量模式”的方法。

来源：[Apple 蓝牙耳机音质原文](../raw/apple/apple-macos-bluetooth-headphones-sound-quality.md)。

## 苹果工程师答复能说明什么

Apple Media Engineer（苹果声音媒体工程师）在 2026 年的一则开发者问答中指出，HFP 是双向传输，会以输入输出成对路径出现；所提的“蓝牙 HFP 输入 + 助听设备输出”组合当前不受支持，并建议提交功能请求。

这份答复来自 iPhone 与 MFi（苹果助听设备认证体系）助听设备场景，不是 macOS 跨设备问题的直接证据。它只能作为架构旁证：苹果声音系统可能把 HFP 当成成对路径处理，不能据此宣称 Apple 已正式确认本项目的 macOS 行为是同一个限制或软件缺陷。

来源：[Apple 开发者问答归档](../raw/apple/apple-developer-forum-hfp-bidirectional-route.md)。

## 第三方工具的实际能力

### 自动改回其他输入

AirPods Sound Quality Fixer（自动把默认麦克风改回其他设备的工具）的项目说明明确写明：它把默认输入强制改为 Mac 内置麦克风，或用户选择的其他输入。既有 Hammerspoon（macOS 自动化工具）方案也采用相同思路。

这类工具可以自动保护蓝牙输出音质，但它解决的是“避免选择蓝牙输入”，不是“选择蓝牙 A 后仍让蓝牙 B 保持 A2DP”。

来源：[AirPods Sound Quality Fixer 项目说明](../raw/community/airpods-sound-quality-fixer-readme.md)、[Hammerspoon 自动切换原文](../raw/community/hammerspoon-airpods-microphone.md)。

### 要求系统不要使用 B 的麦克风

ToothFairy（用于控制蓝牙连接和声音选择的工具）提供“通过禁用设备声音输入改善音质”的选项。其手册称，这会要求 macOS 不要使用该设备的麦克风，从而允许系统采用更适合输出的高质量编码；手册同时说明，应用仍可能覆盖这一选择，再次切回低质量通话编码，之后可能需要断开并重连设备。

对普通的“同一副耳机麦克风被自动选中”问题，这个选项可能有帮助。但跨设备输入输出组合也可能让输出端进入 `tsco`（蓝牙通话传输链路），因此该工具只能列为实验候选，不能列为已验证解决方案。

来源：[ToothFairy 手册原文](../raw/community/toothfairy-disable-audio-input.md)。

## 方案分级

| 方案 | 是否保留蓝牙 A 输入 | B 保持立体声的把握 | 解决层级 | 结论边界 |
| --- | --- | --- | --- | --- |
| A 改用 USB 输入 | 否，A 改走 USB | 高，仍需针对 REDMI 组合实测 | 改变底层连接方式 | 当前优先方案 |
| B 改用以 USB 声音设备呈现的专用接收器 | 是 | 高，取决于接收器能力 | 隔离 B 与 macOS 原生蓝牙路由 | 普通 USB 蓝牙适配器不等于声音接收器 |
| 使用内置或其他 USB 麦克风 | 否 | 已有 `MAC-home01-MacBookAir` 对照支持 | 改变使用条件 | 不能继续使用 A |
| 自动强制选择其他输入 | 否 | 与上一项相同 | 自动化绕过 | 只减少手动操作 |
| 组合声音设备 | 是 | 未知 | 待验证 | 官方资料没有协议隔离承诺 |
| 禁止使用 B 的麦克风 | 是 | 未知，优先级低于 USB 方案 | 待验证的软件干预 | B 麦克风未选中时仍曾降级 |
| 锁定 44.1 kHz、双声道 | 是 | 低 | 只改设备格式 | HFP 重建时格式可能被系统覆盖 |
| 升级系统 | 是 | 未知 | 可能改变系统实现 | 没有找到针对此跨设备现象的明确修复记录 |
| 完整支持 LE Audio（新一代低功耗蓝牙音频）的主机与设备组合 | 可能 | 取决于完整组合 | 更换协议与硬件能力 | 当前 REDMI 链路只有传统蓝牙证据 |

## 当前设备组合的优先候选

系统在 2026-07-17 检测到一个名为 `Redmi 电脑音箱` 的 USB 声音端点，具备 48 kHz（每秒采样四万八千次）输入和输出。其 USB 序列号为 `20190808`，厂商编号为 `0x2717`，产品编号为 `0x5086`。

它是替代蓝牙输入 `Redmi电脑音箱-3899`（蓝牙地址 `50:88:11:07:63:DA`）的首选候选。不过，名称相似不能单独证明两个端点属于同一物理设备；正式验收前应通过拔插或设备指示状态确认映射。

该方案还没有在“USB Redmi 输入实际采集 + REDMI Buds 蓝牙输出持续播放”的完整条件下验收，所以只能写为高优先级候选，不能写成已经解决。


## 不建议直接尝试的改动

- 不把修改系统蓝牙配置文件、删除配对数据库或使用未公开系统接口列为常规方案；现有资料没有证明这些操作可以稳定锁定 B 的协议。
- 不把 Bluetooth Explorer（苹果旧版蓝牙诊断工具）中的编码参数调整解释为 HFP/A2DP 协议锁定；调整编码质量与阻止协议切换不是同一件事。
- 不购买普通通用蓝牙适配器来赌“两个控制器彼此隔离”。只有明确以独立 USB 声音输入或输出设备呈现的接收器，才符合本主题所需的隔离路径。

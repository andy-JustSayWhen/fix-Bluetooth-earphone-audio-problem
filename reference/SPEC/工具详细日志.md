# 工具详细日志

## 文档定位

本文规定蓝牙音频模式检查器自身的详细运行日志，包括默认状态、记录范围、文件格式、轮转、隐私边界和验收方法。

本项目采用 FCMA（按用户功能组织代码的模块化应用架构），详细内容以项目根目录 [`Architecture.md`](../../Architecture.md) 为唯一原文。

## 用户目标

当设备模式刷新延迟、麦克风占用漏报或一键恢复失败时，必须能够还原工具在对应时刻读取了什么、作出了什么判断、执行了哪些步骤，以及每一步返回了什么结果。

## 默认行为

- 详细日志默认启用，不依赖额外启动参数。
- 每次启动必须记录程序版本环境、监听地址、日志文件位置和启动参数。
- 可以通过环境变量 `BLUETOOTH_AUDIO_LOG_ENABLED=0` 临时关闭，但普通启动不得关闭。
- 日志写入失败不得导致设备检查和切换功能崩溃；失败要输出到标准错误。

## 记录范围

至少记录：

1. 服务启动、监听成功、关闭信号和正常退出。
2. 每个网页接口请求的编号、方法、路径、状态码和耗时。
3. 完整设备扫描发生状态变化时的耗时、设备数、默认输入输出，以及扫描失败信息；结果未变化时不重复落盘。
4. 用户手动刷新被合并或限速时，记录跳过原因、距离上次手动扫描的时间和限速阈值，用于排查刷新按钮是否造成高频系统探测。
5. 实时输出事件中的设备名、采样率、声道和运行状态，以及默认输入设备名和运行状态；相同状态不得无间隔重复落盘。
6. 声音输入活动扫描的触发来源、耗时、识别到的进程、是否发生变化、为何继续或停止扫描及读取失败。每个进程必须记录：进程号、进程名、路径、运行输入状态、系统返回的设备列表，以及每个设备的名称、是否为实体输入端点、传输类型、可归属的蓝牙地址、最新 `tacl/tsco` 和链路时间。默认输入开始运行触发时必须明确记录，便于区分“系统事件未到达”“事件到达但输入活动读取失败”和“存在声音输入活动但不属于实体蓝牙麦克风”。
7. 每条声音输入活动必须记录 `inputActivityKind（输入活动类型）`，取值至少区分“已确认占用”“系统声音采集”“未归属声音输入活动”。实体蓝牙麦克风端点关联是占用证据，不要求最新链路为 `tsco`；未闭合 `0 -> 1` 也统一记录为正在占用，同时保留独立的格式请求证据类型和“格式请求”标注，以便与实体端点占用区分。
8. 默认输入输出切换请求、目标、系统是否接受和最终结果。
9. 解除麦克风占用时，记录实体蓝牙麦克风端点关联、被处理进程、已解除进程和仍满足实时占用门槛的进程。不要求最新链路为 `tsco`；没有实体端点关联不得生成解除占用日志。
10. 一键恢复必须按固定顺序逐步记录：实时蓝牙麦克风占用及处理结果、默认输入 A 到非蓝牙 C 再回 A、默认输入输出 A/B 到非蓝牙 C/D 后先回 B 再回 A、格式请求及处理结果、不同蓝牙输入输出与多蓝牙拒绝进程、一次中转输入切换和切回，以及“关闭蓝牙并独立确认状态 → `bluetoothd` → `bluetoothuserd` → `coreaudiod`及蓝牙声音插件 → `audioaccessoryd` → `audiomxd` → 打开蓝牙 → 按需连接目标设备”的完整链路重建和最终稳定验证结果。关闭命令无报错但状态仍为开启时，必须记录真实关闭失败，并且不得记录后续服务已作为本轮完整重建执行。
11. 系统声音日志必须记录查询窗口、每个进程最新的 `0 -> 1` 或 `1 -> 0`、是否形成未闭合请求、请求进程身份及进程号复用检查。未闭合 `0 -> 1` 在页面和单独解除中统一计为带“格式请求”标注的占用；在一键修复中仍由第四步处理，不得混入第一步的实体蓝牙麦克风占用处理。
12. 明确记录本轮未执行结束进程、切换输入输出、重启链路服务、切换蓝牙电源或连接目标设备等动作及原因。
13. 后台子进程的启动、退出码和错误输出。
14. 每台蓝牙输出设备的 A2DP 支持能力及判定依据；输出可用采样率最高值低于 `44.1 kHz` 时必须记录“不支持 A2DP、该设备无需修复”，并记录其是否仍作为其他目标原因判断中的输入端点或链路证据。
15. 扬声器占用状态变化必须记录应用会话、进程号、进程名、输出端点、归属蓝牙地址、是否正在输出和事件时间；没有输出端点或进程已经退出不得记录为当前占用。
16. 一键断开重连必须记录目标设备、请求时的扬声器占用证据、断开结果、重连结果、总耗时和错误；不得把断开成功单独记录成最终修复成功。

## 格式与时间

- 使用 JSONL（每一行都是一条独立 JSON 记录的文本格式）。
- 每条记录包含：时间、级别、进程号、事件名和详细字段。
- 时间使用带时区的 ISO 8601（机器和人都容易排序的标准时间格式）。
- 同一网页请求使用请求编号串联开始与结束。
- 一键恢复的父进程和子进程写入同一日志文件，以设备名和恢复步骤对齐。

## 文件位置与轮转

- 默认目录：`tools/bluetooth-audio-mode-checker/logs/`。
- 当前日志：`app.jsonl`。
- 单文件达到 10 MB 后轮转，最多保留当前文件及 4 份历史文件。
- 日志目录属于本机运行产物，不提交到 Git（代码版本管理工具）。

## 隐私与稳定性边界

- 不记录请求正文的任意原文，只记录本工具明确需要的设备名、方向和进程号等字段。
- 字段名命中密码、令牌、授权头或 Cookie（网站保存登录状态的小数据）时，值必须替换为“已隐藏”。
- 单个文本字段限制长度，避免系统错误或子进程输出无限放大日志。
- 日志不得记录音频内容、麦克风录音或蓝牙传输载荷。
- 高频轮询只记录状态变化、耗时摘要和错误，不逐行复制完整系统报告。

## 查询与页面边界

- 服务提供只读日志状态接口，返回是否启用、当前文件路径、大小和轮转上限。
- 当前版本不在网页内展示或下载完整日志，避免大文件阻塞页面。
- 日志查询不得触发新的完整设备扫描。

## 性能要求

- 日志写入不得把刷新设备和切换设备重新变为同步慢操作。
- 实时声音事件只在状态指纹变化时记录。
- 单条日志写入失败不得中断主流程。
- 手动刷新按钮只能请求合并后的后台扫描；连续点击不得连续启动完整系统设备枚举或反复读取蓝牙输出能力。固定冷却时间只能由服务端刷新调度承担，页面不得用额外倒计时锁住按钮。

## 验收

1. 普通启动后，无需参数即可看到 `logs/app.jsonl`。
2. 调用设备接口、切换默认设备和执行一键恢复后，日志中存在相应请求和步骤记录。
3. 失败恢复必须记录诊断、全部步骤和最终失败采样率。
4. 日志状态接口返回的路径与实际文件一致。
5. 自动测试覆盖默认启用、敏感字段隐藏、长文本截断和轮转配置。
6. 启用日志后重新核验刷新设备与切换设备速度，不得出现明显退化。
7. 默认输出保持内置扬声器、蓝牙设备保持默认输入时，开始一次语音采集；日志必须先出现默认输入运行事件，再出现由该事件触发的声音输入活动扫描。进程和实体蓝牙麦克风端点两段证据齐全时，页面显示麦克风占用，不要求该设备最新链路为 `tsco`。
8. 真实进程关联实体蓝牙麦克风端点时，日志必须记录“已确认实体麦克风占用”，并包含进程和端点两段证据；最新链路作为独立现场事实记录，不参与占用门槛。
9. `replayd` 只关联 `AudioTap` 或设备列表为空时，日志必须记录“系统声音采集”或“未归属声音输入活动”，不得记录为麦克风占用，也不得生成结束进程或授权事件。
10. 两个新增切换步骤必须分别记录 A、B、候选 C、D、候选传输类型、候选优先级、每次切换确认、链路等待、切回次序和最终观察；没有候选时必须明确记录，且日志中不得出现蓝牙中转候选。
11. 当前输入输出来自两台不同蓝牙设备时，日志必须记录设备组合、是否出现 `more than one BT device connected`、能否锁定唯一请求进程、是否结束该进程，以及非蓝牙中转输入切换、链路等待、切回和观察结果。
12. 前五步未恢复时，日志必须依次记录关闭蓝牙，五个服务各自的新旧进程号与重启结果，重新打开蓝牙、目标是否自动连接、是否主动连接、端点重现和路由恢复结果；不得回到前面的步骤循环匹配，也不得再执行一次独立断开重连。
13. 输出可用采样率最高值低于 `44.1 kHz` 的设备不得出现一键修复失败日志；当它影响另一台支持 A2DP 的设备时，日志必须记录受影响设备仍按固定步骤处理。
14. 应用开始和停止向蓝牙输出端点送出声音时，日志必须分别记录扬声器占用建立和清除；只有输入端点、空设备列表或已经退出的进程不得建立归属。
15. 一键断开重连成功日志必须同时包含断开与重连结果，失败日志必须指出失败阶段。

## 实现落点

- 核心日志能力：`tools/bluetooth-audio-mode-checker/core/detailed-logging/`
- 系统日志时间、开机时间与逐行读取能力：`tools/bluetooth-audio-mode-checker/core/macos-unified-log/`
- 服务与接口接入：`tools/bluetooth-audio-mode-checker/app/index.ts`
- 一键恢复接入：`tools/bluetooth-audio-mode-checker/features/a2dp-recovery/`
- 扬声器占用与断开重连接入：`tools/bluetooth-audio-mode-checker/features/speaker-occupancy/`
- 本机日志目录：`tools/bluetooth-audio-mode-checker/logs/`

## 实现后验证

- 自动测试必须覆盖实体蓝牙麦克风占用、输入单独切换、输入输出联合切换、各级非蓝牙候选、无非蓝牙候选、未闭合格式请求、`replayd + AudioTap`、内置输入、空设备列表、双蓝牙且有拒绝进程、双蓝牙但无可确认进程、五个服务逐个重启、蓝牙电源恢复、目标自动连接和目标主动连接。
- 日志必须能按时间还原六个固定处理步骤、每步证据、动作、等待和观察结果，不得只记录最终结果。
- 启用新字段后重新核验刷新设备与切换设备速度，不得出现明显退化。
- 真实一键修复验收必须确认系统声音采集进程不会收到退出请求或阻止授权，格式请求与实时麦克风占用分开处理，两个新增切换步骤和双蓝牙步骤只使用统一优先级选出的非蓝牙设备。

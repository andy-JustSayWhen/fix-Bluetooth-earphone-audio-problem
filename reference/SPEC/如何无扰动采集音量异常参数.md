# 如何无扰动采集音量异常参数

## 文档定位

本文定义蓝牙音量异常排查时的参数采集要求。目标不是根据听感猜测原因，而是在不主动读写音量、不反复创建声音客户端的前提下，记录异常前、异常中和恢复后的同一组参数。

本项目采用 FCMA（按用户功能组织代码的模块化应用架构），详细内容见项目根目录 [`Architecture.md`](../../Architecture.md)。

## 必须记录的参数

每条音量或静音变化事件至少记录：

- 事件时间；
- 当前默认输出设备名称和系统设备编号；
- 触发事件的属性、作用范围和声道编号；
- 系统虚拟主音量是否可读及其数值；
- 设备硬件主音量是否可读及其数值；
- 左右声道等各输出声道的音量；
- 各声道平均音量；
- 静音属性是否可读及其状态；
- 当前标称采样率、实际采样率和运行状态。

事件发生后，应使用同一时间窗口的系统日志补充可能的写入进程、蓝牙模式和播放队列变化。监听器本身无法证明写入者时，必须明确写成“写入者未知”，不得根据时间接近直接归因。

## 无扰动约束

- 使用一个常驻进程注册系统声音属性监听，只在属性变化时读取快照。
- 禁止通过循环启动脚本、辅助程序或设备枚举命令采样。
- 禁止为了采集而写入、微调、同步或恢复音量及静音状态。
- 禁止固定间隔轮询系统音量或设备音量。
- 默认输出变化时，必须注销旧设备监听并注册新设备监听。
- 启动时允许输出一条基线快照；之后只有收到系统属性事件才输出。
- 输出采用逐行 JSON（每一行都是一条完整机器可读记录），便于与系统日志按时间对齐。

## 异常判定边界

以下现象可以记为参数异常，但不能单独证明根因：

- 用户没有执行静音操作时，静音状态发生翻转；
- 同一个事件快照中的系统虚拟主音量与设备声道音量在稳定期持续不一致；不同时间先后读取到的数值不得冒充同时分叉；
- 一次按键后音量向相反方向回摆；
- 默认输出没有改变，但设备编号、声道结构或采样率发生非预期变化；
- 音量异常与输出运行停止、采样率下降或声音队列重建同时出现。

用户正在连续按音量键时，音量连续变化本身不算异常。必须结合用户操作时间、属性事件顺序和系统日志判断。

## 修复验收

不得仅凭命令执行成功或症状暂时消失宣称修复成功。至少需要证明：

1. 与异常窗口相同的复现操作不再产生静音翻转、反向回摆或持续数值分叉；
2. 系统音量与设备各声道音量最终收敛到预期值；
3. 输出设备、采样率、声道数和蓝牙活动模式符合预期；
4. 实际听感异常消失；
5. 采集器没有产生周期性声音客户端注册、设备状态抖动或音频过载。

如果任何一项缺少证据，只能报告当前进展和未决项，不能宣称已经定位根因或完成修复。

# 进入 HFP 模式的原因与恢复实测记录

## 文档用途

本文档是已确证事实库，只记录：

1. 已经通过直接证据确认的 HFP 触发原因；
2. 已经在具体电脑和具体蓝牙设备上实际恢复成功的办法。

本文档不得写入疑似原因、理论可行方法、仅已实现但未实测成功的方法，也不为它们保留占位条目。

本文档是 [`如何一键恢复A2DP模式.md`](如何一键恢复A2DP模式.md) 的已确证经验依据。一键恢复的正式流程和成功标准仍以该规格文档为准。

## 准入标准

### 原因准入

一项原因只有同时满足以下条件才能写入：

1. 能确认目标蓝牙设备当时是活动输出设备；
2. 实际输出采样率从高于 `16 kHz` 变为 `16 kHz` 或更低；
3. 能直接观测到候选原因发生；
4. 时序上是原因先发生，低采样率后出现；
5. 记录具体电脑、设备、日期和证据。

仅有时序相关、程序正在运行或某个设置被选中，不足以写入。

### 恢复办法准入

一项恢复办法只有同时满足以下条件才能写入：

1. 执行前，目标设备的实际输出不高于 `16 kHz`；
2. 记录完整执行步骤；
3. 执行后，目标设备仍是当前默认输出；
4. 至少连续两次读取到实际输出高于 `16 kHz`，两次间隔至少 `500 ms`；
5. 记录具体电脑、设备、日期和结果。

“操作已执行”、“听感变好”、“设备已重连”或“系统已接受请求”均不能单独证明恢复成功。

## 已确认的 HFP 触发原因

### C1：本机程序实际读取目标蓝牙设备的麦克风

- 电脑：当前 macOS 主机。
- 设备：REDMI Buds 6 Pro 竞技版。
- 日期：2026-07-17。
- 直接证据：Codex 语音输入开始后，系统检测到 Codex 正在读取该设备麦克风；同期目标设备的实际输出为 `16 kHz`。
- 可确认结论：在本机与该设备的本次实测中，实际读取蓝牙麦克风会使目标设备进入低采样率输出状态。
- 结论边界：本条不能扩大为“只要设为默认输入就会进入 HFP”。

## 已实测成功的恢复办法

暂无满足本文档准入标准的恢复记录。

已观察到“Codex 语音输入结束后听感恢复”，但当时没有留存间隔至少 `500 ms` 的连续两次高采样率读数，因此不将它写作已实测成功的恢复办法。

## 维护规则

1. 不确证的原因和恢复方法不得写入本文档。
2. 每条记录必须绑定具体电脑和具体蓝牙设备，不得直接扩大为所有 macOS 主机或所有蓝牙设备的通用事实。
3. 没有原始证据、完整操作步骤和采样率验证的结论不得补写。
4. 新证据与已有记录冲突时，不得擅自删除原记录；必须先核对电脑、系统版本、设备、连接方式和实验条件。
5. 本文档新增或删除结论后，必须同步检查一键恢复规格的原因分支、执行顺序和页面措辞。

# 进入 HFP 模式的原因与恢复实测记录

## 文档用途

本文档是已确证事实库，只记录：

1. 已经通过直接证据确认的 HFP 触发原因；
2. 已经在具体电脑和具体蓝牙设备上实际恢复成功的办法。

本文档不得写入疑似原因、理论可行方法、仅已实现但未实测成功的方法，也不为它们保留占位条目。

本文档是 [`如何一键恢复A2DP模式.md`](如何一键恢复A2DP模式.md) 的已确证经验依据。一键恢复的正式流程和成功标准仍以该规格文档为准。

针对“程序没有开始录音，但提交了蓝牙输入格式请求”的新现场排查，先按 [`如何排查程序只请求蓝牙输入格式但未开始录音.md`](如何排查程序只请求蓝牙输入格式但未开始录音.md) 建立证据链；只有满足本文准入标准后，才能新增到本事实库。

## 准入标准

### 原因准入

一项原因只有同时满足以下条件才能写入：

1. 能确认目标蓝牙设备当时是活动输出设备；
2. 实际输出采样率从高于 `16 kHz` 变为 `16 kHz` 或更低；
3. 能直接观测到候选原因发生；
4. 时序上是原因先发生，低采样率后出现；
5. 记录具体电脑、设备、日期和证据。

仅有时序相关、程序正在运行、某个设备被设为系统级或应用级默认输入，或某个设置被选中，不足以写入。默认输入只是候选路由事实，不等于已经进入 HFP；必须同时有活动输出降采样、链路切换、麦克风读取或格式请求等直接证据。

### 恢复办法准入

一项恢复办法只有同时满足以下条件才能写入：

1. 执行前，目标设备的实际输出不高于 `16 kHz`；
2. 记录完整执行步骤；
3. 执行后，目标设备仍是当前默认输出；
4. 至少连续两次读取到实际输出高于 `16 kHz`，两次间隔至少 `500 ms`；
5. 记录具体电脑、设备、日期和结果。

“操作已执行”、“听感变好”、“设备已重连”或“系统已接受请求”均不能单独证明恢复成功。

## 已确认的 HFP 触发原因

### C1：本机程序实际读取目标蓝牙设备的麦克风

- 电脑：当前 macOS 主机。
- 设备：REDMI Buds 6 Pro 竞技版。
- 日期：2026-07-17。
- 直接证据：Codex 语音输入开始后，系统检测到 Codex 正在读取该设备麦克风；同期目标设备的实际输出为 `16 kHz`。
- 可确认结论：在本机与该设备的本次实测中，实际读取蓝牙麦克风会使目标设备进入低采样率输出状态。
- 结论边界：本条不能扩大为“只要设为默认输入就会进入 HFP”。

### C2：程序没有开始录音，但提交了蓝牙输入格式请求

- 电脑：`MAC-home02-MacMini`，Mac mini（M1），macOS 15.7.7（`24G720`）。
- 设备：`Redmi电脑音箱-3899`，蓝牙地址 `50:88:11:07:63:DA`。
- 日期与时间：2026-07-18 09:14:36.187 至 09:14:36.287。
- 直接证据：微信输入法进程检查麦克风权限，并在 09:14:36.254 提交蓝牙音频格式请求；下一毫秒系统执行配置切换，09:14:36.287 输出进入 `tsco`。该窗口没有微信输入法的 `StartIO`。
- 因果对照：退出微信输入法后，于 09:22:32 重复断开重连；默认输入和默认输出仍均为 Redmi，输出连续六次保持 44.1 kHz、双声道和 `tacl`。
- 可确认结论：在本机、本系统版本和本设备组合中，微信输入法不需要真正开始传输麦克风声音；它在该时窗提交的蓝牙输入格式请求已足以触发 HFP。
- 结论边界：2026-07-18 10:00:12 重新启动微信输入法并空闲运行后，约 40 次读取均保持 A2DP，没有新格式请求。因此不能进一步写成“微信输入法一直在空闲请求”或“只要运行就必然触发 HFP”。该格式请求的上游触发条件仍未确认。
- 排查方法：新设备或新电脑出现类似现象时，先按 [`如何排查程序只请求蓝牙输入格式但未开始录音.md`](如何排查程序只请求蓝牙输入格式但未开始录音.md) 重新核对本机身份、目标设备、格式请求和同进程 `StartIO`，不得直接套用本条结论。
- 案例记录：[`../../knowledge/wiki/cases/2026-07-18-微信输入法探测蓝牙输入触发HFP.md`](../../knowledge/wiki/cases/2026-07-18-微信输入法探测蓝牙输入触发HFP.md)。

### C3：选择蓝牙输入 A 触发另一台蓝牙输出 B 降级

- 电脑：`MAC-home02-MacMini`，Mac mini（M1），macOS 15.7.7（`24G720`）。
- 输入 A：`Redmi电脑音箱-3899`，蓝牙地址 `50:88:11:07:63:DA`。
- 输出 B：`REDMI Buds 6 Pro 电竞版`，蓝牙地址 `00:BB:43:41:6B:81`。
- 日期与时间：2026-07-17 08:36:04.120 至 08:36:05.805；持续性验证为 08:38:38.241 至至少 08:39:42。
- 直接证据：把 A 设为默认输入后，B 从 `tacl`、44.1 kHz、双声道切到 `tsco`、16 kHz、单声道；日志持续显示 `AudioInput: 0`，关闭系统设置和控制中心后状态仍维持。
- 可确认结论：在该特定主机和设备组合中，选择蓝牙输入 A 本身足以触发蓝牙输出 B 进入 HFP，不要求应用实际读取 A 的麦克风。
- 结论边界：这是跨设备组合异常，不能推广为“系统级或应用级默认输入本身等于 HFP”，也不能推广为同一蓝牙设备同时作为输入输出时必然进入 HFP；C2 的退出后对照已经证明 Redmi 同时作为默认输入输出时仍可保持 A2DP。
- 案例记录：[`../../knowledge/wiki/cases/2026-07-17-REDMI跨设备蓝牙输入输出联动.md`](../../knowledge/wiki/cases/2026-07-17-REDMI跨设备蓝牙输入输出联动.md)。

## 维护规则

1. 不确证的原因和恢复方法不得写入本文档。
2. 每条记录必须绑定具体电脑和具体蓝牙设备，不得直接扩大为所有 macOS 主机或所有蓝牙设备的通用事实。
3. 没有原始证据、完整操作步骤和采样率验证的结论不得补写。
4. 新证据与已有记录冲突时，不得擅自删除原记录；必须先核对电脑、系统版本、设备、连接方式和实验条件。
5. 本文档新增或删除结论后，必须同步检查一键恢复规格的原因分支、执行顺序和页面措辞。
